# 多選日期功能 - 回滾機制說明

## 📋 概述

本文檔詳細說明多選日期功能的回滾機制、使用方法和相關風險。

---

## 🔄 回滾機制

### 什麼是回滾？

回滾是一個**安全恢復機制**，用於將市集的日期資料重置為連續日期模式。

### 回滾做了什麼？

```typescript
// 回滾前（多選日期）
dates: ['2024-02-15', '2024-02-16', '2024-02-22', '2024-02-23']

// 回滾後（連續日期）
dates: ['2024-02-15', '2024-02-16', '2024-02-17', '2024-02-18', 
        '2024-02-19', '2024-02-20', '2024-02-21', '2024-02-22', '2024-02-23']
```

回滾會根據 `startDate` 和 `endDate` 重新生成**連續的日期陣列**。

---

## ⚠️ 風險評估

### 🔴 高風險操作

**資料丟失風險**：
- ✅ **會丟失**：用戶手動選擇的不連續日期
- ✅ **會丟失**：特定日期的選擇邏輯（例如：只選週末）
- ❌ **不會丟失**：收入、成交、互動等統計資料
- ❌ **不會丟失**：每日統計資料
- ❌ **不會丟失**：市集的其他資訊

### 🟡 中風險操作

**顯示影響**：
- 市集卡片的日期顯示會改變
- 從「2024-02-15~16, 2024-02-22~23」變為「2024-02-15~23」

**統計影響**：
- 每日統計不受影響（仍然按實際有資料的日期顯示）
- 總計統計不受影響

### 🟢 低風險操作

**不受影響的部分**：
- ✅ 市集名稱、地點、狀態
- ✅ 財務資訊（攤位費、租金等）
- ✅ 時間軸設定
- ✅ 收入和成交記錄
- ✅ 商品資料
- ✅ 每日統計

## 🎯 使用場景

### ✅ 適合使用回滾的情況

1. **資料損壞**
   - dates 陣列為空或格式錯誤
   - startDate/endDate 與 dates 不一致

2. **測試環境重置**
   - 開發測試時需要重置資料
   - 清理測試資料

3. **誤操作恢復**
   - 用戶不小心選錯日期
   - 想要恢復為連續日期模式

### ❌ 不適合使用回滾的情況

1. **正常的多選日期**
   - 用戶刻意選擇不連續日期（例如：只有週末）
   - 這種情況下回滾會破壞用戶的選擇

2. **資料正常運作**
   - 如果系統運作正常，不需要回滾

## 🔍 技術細節

### 回滾邏輯

```typescript
export async function rollbackDatesField(): Promise<void> {
  // 1. 讀取所有市集
  const markets = await db.markets.toArray();
  
  // 2. 逐一處理
  for (const market of markets) {
    // 3. 生成連續日期
    const dates = generateDateRange(market.startDate, market.endDate);
    
    // 4. 更新資料庫
    await db.markets.update(market.id!, { dates });
  }
}
```

### 資料完整性保證

- ✅ 使用 Dexie 的 `update` 方法，只更新 `dates` 欄位
- ✅ 不會影響其他欄位
- ✅ 原子性操作，不會出現部分更新
- ✅ 錯誤處理，失敗時不會破壞資料

## 🆘 常見問題

### Q1: 回滾會刪除我的收入記錄嗎？

**A**: 不會。回滾只會修改市集的 `dates` 欄位，不會影響收入、成交、互動等記錄。

### Q2: 回滾後可以恢復嗎？

**A**: 不可以直接恢復。但如果你在回滾前備份了資料，可以通過「匯入資料」功能恢復。

### Q3: 什麼時候應該使用回滾？

**A**: 只在以下情況使用：
- 資料損壞或不一致
- 測試環境需要重置
- 確定要放棄多選日期，改用連續日期

### Q4: 回滾會影響其他用戶嗎？

**A**: 如果是本地資料，只影響當前裝置。如果啟用了雲端同步，會影響所有同步的裝置。

### Q5: 回滾失敗怎麼辦？

**A**: 回滾失敗不會破壞資料。可以：
1. 查看控制台錯誤訊息
2. 重新整理頁面再試一次
3. 如果持續失敗，使用備份恢復
