## 1. 問題描述

### 1.1 Fail-Open 風險

`useUserRole()` 初始值為 `{ isStaff: false }`（owner 預設）；`resolveInfoLevel(userRole)` 對 owner 回傳 `3`（無限制脫敏）。

**情境**：

| 情境 | 原行為（fail-open） | 修正後（fail-closed） |
|---|---|---|
| `useUserRole` 初始化（useState 初值） | `userRole={isStaff:false}` → `infoLevel=3` → `isDataSanitized=false` | `isLoading=true` → `deriveSafeInfoLevel=0` → `isDataSanitized=true` |
| Supabase 查詢中（`isLoading=true`） | `infoLevel=3` → useSync 寫入**全部欄位**到員工 IndexedDB | `infoLevel=0` → useSync 寫入時過濾敏感欄位 |
| Supabase 查詢失敗（catch） | `setUserRole({isStaff:false})` → `infoLevel=3` | `setUserRole({isStaff:true, can_view:false, can_edit:false})` + `setRoleError(err)` → `infoLevel=0` |
| 未登入 | `infoLevel=3` | `infoLevel=0` |

**攻擊面**：

- 員工首次登入 / 新裝置登入 / Supabase 暫時不可用期間，**會把 owner 的 cost/profit/supplier 敏感欄位寫進員工 IndexedDB**
- 這違反 C2.30C/D 與 C2.31 三層脫敏防線

### 1.2 既有呼叫點的 fail-open 行為

| 檔案 | 引用 | 風險 |
|---|---|---|
| `lib/sync-context.tsx:33-36` | `useUserRole` + `resolveInfoLevel` | 🟠 **P0**：infoLevel 在 loading 期間 = 3 |
| `hooks/useUserRole.ts:185-186` | catch fallback `{ isStaff: false }` | 🟠 **P0**：錯誤期間被當 owner |

## 2. 修補設計

### 2.1 純函式（`lib/permissions/role-fail-closed.ts`）

```ts
export interface RoleSnapshot {
  userRole: UserRole | null | undefined;
  isLoading: boolean;
  roleError: Error | null;
}

export function deriveRolePermissions(snapshot: RoleSnapshot): RolePermissions {
  const isStaff = snapshot.userRole?.isStaff ?? false;
  const isResolved = !snapshot.isLoading && !snapshot.roleError && snapshot.userRole != null;
  const isOwner = isResolved && !isStaff;
  return {
    isOwner,
    canEdit: isOwner,
    canViewSensitiveData: isOwner,
    isStaff,
  };
}

export function deriveSafeInfoLevel(snapshot: RoleSnapshot): InfoLevel {
  if (snapshot.isLoading || snapshot.roleError || snapshot.userRole == null) {
    return 0; // 最嚴格
  }
  if (!snapshot.userRole.isStaff) return 3; // owner
  return (snapshot.userRole.permissions as { infoLevel?: InfoLevel } | null)?.infoLevel ?? 2;
}

export function isRoleInFailOpenState(snapshot: RoleSnapshot): boolean {
  return snapshot.isLoading || snapshot.roleError != null || snapshot.userRole == null;
}
```

## 3. 測試覆蓋

### 3.1 `tests/role-fail-closed.test.ts`（18 個測試）

#### `deriveRolePermissions`（7 個）

- ✅ owner 已載入 → 全 true
- ✅ staff 已載入 → 全 false，isStaff 為 true
- ✅ role loading 期間 → 全 false（fail-closed）
- ✅ role 查詢錯誤 → 全 false（fail-closed）
- ✅ userRole 為 null → 全 false
- ✅ userRole 為 undefined → 全 false
- ✅ loading + error 同時 → 全 false

#### `deriveSafeInfoLevel`（7 個）

- ✅ owner 已載入 → infoLevel 3
- ✅ staff 已載入無 infoLevel → infoLevel 2
- ✅ staff with permissions.infoLevel=1 → infoLevel 1
- ✅ role loading → infoLevel **0**（fail-closed）
- ✅ role error → infoLevel **0**（fail-closed）
- ✅ userRole null → infoLevel **0**
- ✅ loading + error → infoLevel **0**

#### `isRoleInFailOpenState`（4 個）

- ✅ loading=true → true
- ✅ roleError != null → true
- ✅ userRole=null → true
- ✅ 全部確認後 → false

## 4. 影響範圍（向後相容性）

### 4.1 既有 25 個 `useUserRole` 呼叫點

**完全不需要改**：

- 語意 `isStaff` 仍從 `userRole.isStaff` 讀
- 語意 `isOwner` / `canEdit` / `canViewSensitiveData` 改為**更嚴格**（loading/error 期間 false 而非 true）
- 沒有任何呼叫點依賴「loading 期間 = owner」這個 fail-open 行為

**唯一行為差異**（更安全）：

- loading 期間：
  - 原本：`isOwner=true` / `canEdit=true` / `canViewSensitiveData=true`
  - 現在：`isOwner=false` / `canEdit=false` / `canViewSensitiveData=false`
- 錯誤期間：
  - 原本：catch fallback `{ isStaff: false }` → 視為 owner
  - 現在：`roleError != null` + `{ isStaff: true, can_view: false, can_edit: false }` → 視為最嚴格員工

### 4.2 `useSync` 主流程

**未動**：

- `useSync.ts` 主流程未改
- `useSync` 的 `enabled` 仍由 `!isRoleLoading` 控制（已存在）
- `roleInfoLevel` 參數仍由 SyncProvider 傳入（值從 3 變 0，更安全）

### 4.3 `PermissionGate` 與 `resolveInfoLevel`

**未動**：

- `PermissionGate.ts` 的 `resolveInfoLevel(userRole)` 仍存在（給予 UI 層使用）
- 新增的 `deriveSafeInfoLevel(snapshot)` 是 fail-closed 版本（給予 sync-context 等 critical path 使用）
- 兩個函式可共存；UI 層用 `resolveInfoLevel`，同步層用 `deriveSafeInfoLevel`
