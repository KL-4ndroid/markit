## 0. 為何需要 C2.29B-2.1

### 0.1 殘留攻擊面（E2 線上實測確認）

039 修了 view 層、040 修了 view scope，但**底表 RLS 仍放行員工**：

| 攻擊面 | 039 + 040 修法 | 線上實測 |
|---|---|---|
| **#A** Staff 透過底表 RLS SELECT `markets` 拿到 `booth_cost` / `total_profit` | ❌ 沒修 | ✅ **E2 確認**（員工透過 RLS 拉到 `booth_cost=0.00`）|
| **#B** Staff 透過底表 RLS SELECT `products` 拿到 `cost` | ❌ 沒修 | 🟠 預期成立（products `team` 政策從 014 就放行）|
| **#C** Staff 透過底表 RLS SELECT `events` 拿到完整 `payload` | ❌ 沒修 | 🟠 預期成立（events `market_members` 政策從 015 就放行）|

### 0.2 攻擊面根因

**`markets_select_secure`（035）+ `Users can view own and team products`（014）+ `用戶可以查看自己的事件和市集事件`（015）**全部透過 `market_members` 放行員工：

- 員工是市集成員（021 trigger 自動加 `role = 'staff'`）→ 命中 `current_user_market_ids()` → 命中 markets RLS
- 員工透過 `team` 路徑（products 014）拿到 products
- 員工透過 `market_members` 路徑（events 015）拿到 events

→ **員工 SELECT 底表 = 0 row 過濾失敗**。

### 0.3 041 解法

**Owner 判斷採「事實欄位」`m.owner_id` / `p.owner_id` / `e.actor_id`**（不依賴 021 / 002 trigger 維護的 `market_members`）：

| 表 | 修法 | 員工結果 |
|---|---|---|
| `markets` | 刪 `markets_select_secure`，新增 `markets_select_owner_only`（`owner_id = auth.uid()`）| 0 row |
| `products` | 刪 `Users can view own and team products`（含 `team` + `is_shared` 路徑），新增 `products_select_owner_only`（`owner_id = auth.uid()`）| 0 row |
| `events` | 刪 2 條線上 SELECT policy（015 + `users_can_view_events`），新增 `events_select_owner_only`（`market_id IN owned` OR `market_id IS NULL AND actor_id = self AND EXISTS owner`）| 0 row |

### 0.4 events policy 設計決策（v2 修正，2026-06-16）

**初版設計**：

```sql
-- ❌ 初版：仍放行 staff
USING (
  actor_id = auth.uid()  -- 員工自己寫的事件（透過 025 staff_relationships）
  OR
  market_id IN (SELECT id FROM public.current_user_owned_market_ids())
)
```

**問題**：員工透過 `events` INSERT policy 025（員工可插入老闆市集的事件）寫入的 event，會命中 `actor_id = auth.uid()` 路徑，**員工仍可 SELECT 自己寫的事件**。這違反 C2.29B-2.1 「Staff direct SELECT = 0 row」的核心目標。

**修正版（v2，已套用至 041）**：

```sql
-- ✅ 修正：owner-only 全局事件需綁定「自己是某市集 owner」
USING (
  market_id IN (SELECT id FROM public.current_user_owned_market_ids())
  OR
  (
    market_id IS NULL
    AND actor_id = auth.uid()
    AND EXISTS (SELECT 1 FROM public.markets m WHERE m.owner_id = auth.uid())
  )
)
```

**語意**：

- ✅ Owner 旗下市集事件：可讀（market_id IN owned）
- ✅ Owner 自己寫的全局事件（product_created 等 `market_id IS NULL`）：可讀（綁定 owner 條件）
- ❌ **Staff 自己寫的事件（含 `market_id IS NULL` 與市集綁定兩種）**：EXISTS 條件不成立 → 0 row
- ❌ **Staff 透過 025 staff_relationships 寫入的事件**：EXISTS 不成立 → 0 row
- ❌ **員工 actor_id = auth.uid() 任意事件**：EXISTS 不成立 → 0 row（**v2 修正重點**）

> **注意**：這條 policy 比 `actor_id = auth.uid()` 更嚴格。若 owner 透過 view 拉 `staff_accessible_events` Branch 3（`actor_id = auth.uid()`），會在 view 內部用 caller auth.uid() 跑查詢，**底表 RLS 收緊後**需要 view 為 SECURITY DEFINER 才可繞過（見 §0.5）。

### 0.5 view security mode 需求（CRITICAL）

**PostgreSQL 15+ 預設 `security_invoker = true`**。如果 `staff_accessible_*` view 是 `SECURITY INVOKER`：

- Staff 透過 `staff_accessible_markets` 查詢：view 內部 `FROM markets m` 觸發 `markets_select_owner_only`（`owner_id = auth.uid()`），**staff 命中 0 row** → view 完全失效
- Owner 透過 `staff_accessible_events` Branch 3（`actor_id = auth.uid()`）：底表 events RLS 改用 EXISTS 綁 owner 後，**market_id IS NULL 全局事件可能被擋**

→ **若 view 是 INVOKER，041 不可直接套用**，需先建立 042 把 view 改為 DEFINER（**042 不在本次範圍**）。

**041 套用前必跑 preflight**（P0.3 / P0.4）：

```sql
SELECT
  c.relname AS view_name,
  CASE
    WHEN c.reloptions IS NULL THEN 'security_definer (default for PG14-created views)'
    WHEN c.reloptions::text LIKE '%security_invoker%' THEN 'SECURITY INVOKER (BREAKS 041)'
    ELSE 'security_definer (custom options)'
  END AS security_mode,
  c.reloptions
FROM pg_class c
JOIN pg_namespace n ON n.oid = c.relnamespace
WHERE n.nspname = 'public'
  AND c.relname IN (
    'staff_accessible_markets',
    'staff_accessible_products',
    'staff_accessible_events'
  )
ORDER BY c.relname;
```

`041` 內建 DO 區塊（P0.4）會在 INVOKER 偵測到時 `RAISE EXCEPTION` 中止套用。
