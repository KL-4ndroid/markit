## 1. Root Cause（用戶觀察 + Cursor 程式碼靜態分析確認）

### 1.1 用戶 SQL 觀察（2026-06-16）

用戶在 Staff 視角下補跑以下 SQL 後發現：

1. **`staff_accessible_markets` 同一 market id 同時出現 staff + owner branch**
2. **多個 `is_deleted = true` market 仍被 Staff 拉進來**
3. **`staff_accessible_events` `market_id IS NULL` 事件中**：
   - `product_deleted`: 13 個
   - `product_created`: 8 個
4. **前端 useSync log 印出**：
   ```
   Skipping event outside local scoped dataset
   reason: missing_market_id
   eventType: product_created
   ```

### 1.2 Cursor 程式碼靜態分析（039 全文確認）

#### 1.2.1 Bug #A：staff_accessible_markets Owner Branch scope leak

**`039_staff_view_hardening.sql` L284-286**：

```sql
-- Branch 2: OWNER（完整欄位）
...
FROM (markets m
    JOIN market_members mm ON ((mm.market_id = m.id)))
WHERE (mm.user_id = auth.uid());
```

**問題**：

- 員工在 `market_members` 表中**有 row**（因為 `021_auto_add_staff_to_markets.sql` 的 trigger 自動加 `role = 'staff'`）
- Owner branch 用 `mm.user_id = auth.uid()` **沒檢查 `mm.role`**
- **員工會命中 Owner branch**，access_type = 'owner'，**拿到完整 `booth_cost` / `total_profit` / `commission_rate` 等**
- → **039 staff branch 脫敏完全被繞過** 🟠 **P0 攻擊面**

**Cursor 對應 SQL 驗證（用戶可手動跑）**：

```sql
-- 套用 039 後，用員工 session 跑
BEGIN;
SET LOCAL ROLE authenticated;
SELECT set_config('request.jwt.claim.sub', '5e92b457-1eaf-49eb-9295-ba47b5a3e575', true);

SELECT
  access_type,
  count(*)
FROM staff_accessible_markets
GROUP BY access_type;
-- 預期（C2.29B-1 套用後）：應只有 staff 1 row
-- 實際（用戶回報）：staff + owner 都出現，access_type='owner' 的 row 數 = 該員工透過 trigger 加入的市集數
ROLLBACK;
```

#### 1.2.2 Bug #B：staff_accessible_events Branch 4 scope leak

**`039_staff_view_hardening.sql` L455-457**：

```sql
-- Branch 4: OWNER（市集成員事件，payload 完整）
...
FROM (events e
    JOIN market_members mm ON ((mm.market_id = e.market_id)))
WHERE ((mm.user_id = auth.uid()) AND (e.actor_id <> auth.uid()));
```

**問題**：

- 同樣的 bug：員工在 `market_members` 有 row（`role = 'staff'`）
- 員工會命中 Branch 4，access_type = 'owner'，**拿到完整 events.payload**
- → 039 staff branch 1 的 `sanitize_staff_event_payload()` 對 Branch 4 沒生效 🟠 **P0 攻擊面**

**Cursor 對應 SQL 驗證**：

```sql
BEGIN;
SET LOCAL ROLE authenticated;
SELECT set_config('request.jwt.claim.sub', '5e92b457-1eaf-49eb-9295-ba47b5a3e575', true);

SELECT
  type,
  payload->>'boothCost' AS booth_cost
FROM staff_accessible_events
WHERE access_type = 'owner'
  AND type = 'market_created'
LIMIT 5;
-- 預期（040 套用後）：0 rows
-- 實際（用戶回報）：員工能透過 Branch 4 拿到真實 boothCost
ROLLBACK;
```

#### 1.2.3 Bug #C：is_deleted 沒過濾

**4 個 view 結構都沒過濾 `is_deleted`**：

- `staff_accessible_markets` Staff branch L237-240
- `staff_accessible_markets` Owner branch L284-286
- `staff_accessible_events` Branch 1 L396-399
- `staff_accessible_events` Branch 4 L455-457

**影響**：

- 軟刪除的市集（`is_deleted = true`）仍會被拉進員工 sync
- 前端會 show 刪除的市集（可能造成 UX 問題）

**Cursor 對應 SQL 驗證**：

```sql

SET LOCAL ROLE authenticated;
SELECT set_config('request.jwt.claim.sub', '5e92b457-1eaf-49eb-9295-ba47b5a3e575', true);

SELECT count(*) AS deleted_markets_in_view
FROM staff_accessible_markets
WHERE COALESCE(is_deleted, false) = true;
-- 預期（040 套用後）：0
-- 實際（用戶回報）：> 0
ROLLBACK;
```

#### 1.2.4 Bug #D：Global product events 仍命中

**`staff_accessible_events` Branch 2（L416-420）**：

```sql
-- Branch 2: STAFF（全局事件）
...
FROM (events e
    JOIN staff_relationships sr ON (sr.owner_id = e.actor_id))
WHERE ((sr.staff_id = auth.uid())
  AND (sr.status = 'active'::text)
  AND (e.market_id IS NULL))
```

**問題**：

- 員工會拉到 `market_id IS NULL` 的 `product_created` / `product_deleted` 等
- 這些事件因為 `market_id IS NULL`，**前端的 `staff-event-preflight.ts` 會以 `missing_market_id` 跳過**（這就是用戶 log 看到的）
- 但**仍佔用 view 拉取的頻寬** + 員工能從 `actor_id` 推測 owner 身份（隱私）

**影響**：

- 🟡 P1 噪音（不是 P0 攻擊面，因為 payload 已被 `sanitize_staff_event_payload()` 處理過）
- 040 **保守不刪** Branch 2：避免破壞既有 event 同步行為
- 改在 Branch 2 結構上做改善（加 `m.is_deleted` 過濾 + 加註解）

**Cursor 對應 SQL 驗證**：

```sql

SET LOCAL ROLE authenticated;
SELECT set_config('request.jwt.claim.sub', '5e92b457-1eaf-49eb-9295-ba47b5a3e575', true);

SELECT type, count(*)
FROM staff_accessible_events
WHERE market_id IS NULL
GROUP BY type;
-- 預期：product_created + product_deleted 仍會出現（Branch 2 保留）
-- 040 套用後數量不變，但前端的 missing_market_id skip 仍會處理
ROLLBACK;
```

### 1.3 為何會有此 bug（線上觀察 → 程式碼分析）

| 維度 | 說明 |
|---|---|
| **歷史** | 此 bug **從 030_fix_data_isolation.sql 就存在**（L107-108 的 Owner branch）。039 沒修 WHERE 條件，只改 SELECT 列表 |
| **039 設計疏漏** | 039 設計時只關注「staff branch 敏感欄位脫敏」，沒驗證「員工是否會誤命中 owner branch」 |
| **Post-Apply Smoke Test 不足** | 039 套用後的驗證 SQL（Section 5）只驗證「敏感欄位是否為 NULL」，**沒驗證「access_type 是否只有 staff」**。這是 smoke test 框架的盲點 |
| **為何現在才發現** | 039 套用後 UI 仍可正常運作（PermissionGate 過濾），但 sync 拉取量大增（37 個市集 + 587 個事件，預期應更少）。員工 DevTools 才會發現 access_type 混用 |

## 2. 為何 Staff 命中 Owner Branch

### 2.1 `market_members` 表的 membership 設計

`market_members` 表（`001_uuid_schema.sql` L116-122）：

```sql
CREATE TABLE IF NOT EXISTS market_members (
  market_id UUID NOT NULL REFERENCES markets(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  role TEXT NOT NULL CHECK (role IN ('owner', 'staff')),
  ...
```

每個市集有「成員」概念，**owner + staff 都會被加進 `market_members` 表**：

| 加入方式 | 觸發器 | role |
|---|---|---|
| Owner 透過 `market_created` 事件 | `002_cqrs_triggers.sql` L82-85 | `'owner'` |
| Staff 透過 owner 新建市集 | `021_auto_add_staff_to_markets.sql` L11-20 | `'staff'` |

### 2.2 Owner branch 條件太寬

039 Owner branch（markets）：

```sql
FROM (markets m JOIN market_members mm ON mm.market_id = m.id)
WHERE (mm.user_id = auth.uid());
```

**問題**：

- 員工在 `market_members` 有 row（`role = 'staff'`，由 021 trigger 自動加入）
- `mm.user_id = auth.uid()` **命中員工 row**
- **完全沒檢查 `mm.role`**

→ 員工會進入 owner branch，access_type = 'owner'，拿到完整敏感欄位

### 2.3 為何 039 沒抓到

039 設計時**只驗證了「欄位輸出是否 NULL」**（Section 5 驗證 1-5），**沒驗證「access_type 分佈是否正確」**。這是 smoke test 框架的盲點。

建議：**未來 view 變更後的 smoke test 必加 `GROUP BY access_type` 驗證**。

## 4. Deleted Markets 來源

### 4.1 軟刪除機制

從 002 trigger 看，市集是透過 `market_deleted` 事件軟刪除：

```sql
WHEN 'market_deleted' THEN
  UPDATE markets
  SET is_deleted = TRUE, ...
```

### 4.2 為何 view 沒過濾

**039 view 結構完全沒過濾 `is_deleted`**，導致已刪除的市集仍會被拉進員工 sync：

- `staff_accessible_markets` Staff branch：L237-240 無 `is_deleted = false` 過濾
- `staff_accessible_markets` Owner branch：L284-286 無 `is_deleted = false` 過濾
- `staff_accessible_events` Branch 1：L396-399 無 JOIN markets 取 `is_deleted`
- `staff_accessible_events` Branch 4：L455-457 無 JOIN markets 取 `is_deleted`

### 4.3 影響

- 員工 UI 看到已刪除的市集（隱私問題：刪除的市集可能含敏感營收資料）
- sync 拉取量增加（噪音）

### 4.4 040 修法

- Staff branch：加 `AND COALESCE(m.is_deleted, false) = false`
- Owner branch：保留（Owner 應可看到自己刪除的市集歷史）
- Staff events Branch 1：JOIN `markets m` + 加 `is_deleted` 過濾
- Staff events Branch 4（owner）：保留（Owner 應可看到完整事件歷史）

## 5. Global Product Events 來源

### 5.1 為何 product_created / product_deleted 會有 `market_id IS NULL`

從 `002_cqrs_triggers.sql` 看，product 觸發器從 `events.market_id` 讀取 market_id：

```sql
-- 002 L143-161
INSERT INTO products (
  id, market_id, name, ...
) VALUES (
  (NEW.payload->>'productId')::UUID,
  NEW.market_id,  -- ← 從 events.market_id 讀取
  ...
);
```

如果 `events.market_id IS NULL`，則 product 同步寫入時 `market_id = NULL`。

### 5.2 為何有 `events.market_id IS NULL` 的 product_created

推測：員工在 Owner 視角下建立「獨立商品」（未綁定市集的商品），前端 `recordEvent('product_created', payload)` 沒帶 `market_id`。

從 002 沒看到 product_created 必須有 market_id 的約束（不像 `market_created` L17 強制要求 marketId）：

```sql
-- 002 L16-18: market_created 強制要 marketId
IF (NEW.payload->>'marketId') IS NULL THEN
  RAISE EXCEPTION 'market_created event missing marketId in payload';
END IF;

-- 002 L138-141: product_created 強制要 productId，但 marketId 是 optional
IF (NEW.payload->>'productId') IS NULL THEN
  RAISE EXCEPTION 'product_created event missing productId in payload';
END IF;
-- 沒強制 productId 必須有 marketId
```

→ **既有 schema 設計允許 product events 沒有 market_id**。040 改不了這個（屬於資料模型層）。

### 5.3 員工 Branch 2 為何命中

039 Branch 2（L416-420）：

```sql
FROM (events e
    JOIN staff_relationships sr ON (sr.owner_id = e.actor_id))
WHERE ((sr.staff_id = auth.uid())
  AND (sr.status = 'active'::text)
  AND (e.market_id IS NULL))
```

**條件**：

- `e.actor_id` 是 owner（owner 建立了這個 product event）
- `sr.staff_id` 是員工
- `sr.owner_id = e.actor_id`（員工綁定的 owner = event 發起人）
- `e.market_id IS NULL`（全局事件）

→ **員工會拉到 owner 旗下的全局 product events**

### 5.4 影響

- 🟡 P1 噪音（不是 P0 攻擊面，因為 payload 已被 `sanitize_staff_event_payload()` 處理過）
- 前端 `staff-event-preflight.ts` 會以 `missing_market_id` 跳過（已知行為）
- 但仍佔用 view 拉取頻寬

### 5.5 040 處理策略（保守）

- **不刪 Branch 2**（避免破壞既有同步行為）
- **不過濾 product_created / product_deleted 數量**（保留現狀）
- 改在 Branch 2 結構上**加註解說明風險**（040 §Section 3 Branch 2）
- 評估是否在**C2.29B-2** 改用 `e.actor_id = m.owner_id JOIN staff`（徹底過濾）

### 5.6 替代方案（**未實作，僅記錄**）

#### 5.6.1 方案 A：刪 Branch 2

- 員工完全看不到全局 product events
- 風險：如果日後 owner 用 product events 傳「非市集綁定」的 metadata，員工會看不到
- 評估：低（產品的 use case 應讓 product 綁定 market）

#### 5.6.2 方案 B：Branch 2 過濾 product_created / product_deleted

- 員工只看全局非 product 事件（cost_added / inventory_cost_updated 等）
- 風險：產品的「非綁定市集商品」概念消失
- 評估：中

#### 5.6.3 方案 C：Branch 2 加 is_active 過濾

- 員工只看 `p.is_active = true` 的商品事件
- 仍會拉 13 + 8 = 21 個
- 評估：低（幾乎沒改善）

→ **040 採方案「保守不動」**（保留 Branch 2 結構）

## 7. 040 驗證 SQL 摘要

完整 SQL 在 `040_fix_staff_accessible_view_scope.sql` Section 4。摘要：

### 7.1 驗證 1：Staff 不應命中 owner branch（scope leak 防護）

| 檢查項 | 預期 |
|---|---|
| `GROUP BY access_type` | 只有 1 row：`access_type='staff'` |
| `GROUP BY id HAVING count(*) > 1` | 0 rows（無重複 market）|
| `WHERE is_deleted = true` | 0 rows（無軟刪除）|

### 7.2 驗證 2：Staff events 不應命中 owner branch

| 檢查項 | 預期 |
