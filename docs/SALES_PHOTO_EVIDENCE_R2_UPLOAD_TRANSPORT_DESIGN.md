# Sales Photo Evidence R2 Upload Transport Design

Date: 2026-07-08
Status: design-only. This document chooses the first R2 upload transport for sales photo evidence. It does not add an R2 client, install an SDK, upload files, issue signed URLs, delete local payloads, enable production runtime enqueue, or mutate production data.

## Decision

Use a server-side `multipart/form-data` route first.

Recommended route:

```txt
POST /api/sales-photo-evidence/upload
```

The browser sends one captured image blob and one thumbnail blob to the existing upload route. The route authenticates the user, reuses the metadata claim adapter/repository, uploads both private objects to R2 through a future injected server-only R2 adapter, then finalizes the Supabase metadata row as `uploaded`.

## Why Not Pre-Signed Upload First

Pre-signed upload URLs are useful later, but they are not the best first transport for this project.

Reasons:

- they require an extra server round trip before upload;
- the browser needs to coordinate object-key, upload, and metadata finalize states;
- failed browser uploads can leave object/metadata mismatch;
- retry logic becomes split across client, R2, and Supabase;
- it increases orphan object risk before we have cleanup tooling.

For the first production-quality path, the safer boundary is one server route that owns the whole upload sequence.

## First Transport Request Shape

The route should accept `multipart/form-data`.

Text fields:

- `ownerId`
- `marketId`
- `saleEventId`
- `capturedByStaffId` as a UUID string or empty string
- `capturedAt`
- `imageMetadata` as JSON
- `thumbnailMetadata` as JSON
- optional `queueId` for client-side local correlation only

File fields:

- `image`
- `thumbnail`

`queueId` must not be trusted for authorization and must not be stored as the source of truth. The source of truth remains Supabase auth, sale event ownership, market ownership, staff relationship, and the active `sale_photo_evidence` row.

## Required Validation

Before any R2 upload:

1. Auth must resolve from the request Bearer token.
2. Route feature gate must be enabled.
3. Request must include exactly one image and one thumbnail.
4. Mime type must be accepted by `createSalesPhotoEvidenceUploadContract()`.
5. File size must be within `SALES_PHOTO_EVIDENCE_COMPRESSION_POLICY.maxFileSizeBytes`.
6. Metadata dimensions must be positive integers.
7. Metadata claim must succeed and return an uploadable evidence row.
8. Object keys must be generated by the existing object-key builder, not supplied by the client.

## Server Execution Order

The server route must keep this order:

1. Authenticate actor.
2. Parse and validate `FormData`.
3. Create or reuse metadata claim as `uploading`.
4. Create upload contract from the claimed evidence row and payload metadata.
5. Upload image object to R2.
6. Upload thumbnail object to R2.
7. Finalize Supabase metadata row as `uploaded`.
8. Return success telling the client it may delete the local payload.

The route must not mark `uploaded` before both R2 uploads succeed.

## Failure Handling

Failure handling must keep local payloads unless upload is already complete.

- Validation failure: no R2 upload, keep local payload.
- Permission failure: no R2 upload, keep local payload.
- Metadata claim failure: no R2 upload, keep local payload.
- Image upload failure: mark row `upload_failed` if possible, keep local payload, do not upload thumbnail.
- Thumbnail upload failure: mark row `upload_failed` if possible, keep local payload, record orphan image as future cleanup candidate.
- Metadata finalize failure: keep local payload, mark row `upload_failed` if possible, record both objects as future cleanup candidates.
- Already uploaded row: return idempotent success and allow local payload deletion.

No automatic R2 deletion or cleanup executor is part of this transport slice.

## Future R2 Adapter Boundary

The first R2 adapter should be server-only and dependency-injected.

Interface shape:

```ts
type SalesPhotoEvidenceR2UploadObjectInput = {
  key: string;
  body: Blob | ArrayBuffer | Uint8Array;
  contentType: 'image/webp' | 'image/jpeg';
  contentLength: number;
};

type SalesPhotoEvidenceR2UploadObjectResult =
  | { ok: true; key: string; etag?: string }
  | { ok: false; key: string; code: 'r2_upload_failed'; message: string };
```

Rules:

- R2 credentials stay server-only.
- No `NEXT_PUBLIC_R2_*`.
- No public bucket URL.
- No signed read URL in this upload slice.
- No R2 client import in browser components or local pending payload storage.
- Tests may use a fake adapter; real SDK wiring requires a separate slice.

## Feature Gate

Metadata claim route enablement is now allowed for local/staging with:

```txt
SALES_PHOTO_EVIDENCE_METADATA_CLAIM_ROUTE_ENABLED=1
```

Production must require a second explicit flag:

```txt
SALES_PHOTO_EVIDENCE_METADATA_CLAIM_ROUTE_ALLOW_PRODUCTION=1
```

R2 upload execution must have its own future gate. Do not reuse the metadata claim route gate as the R2 upload gate.

Recommended future gate:

```txt
SALES_PHOTO_EVIDENCE_R2_UPLOAD_ROUTE_ENABLED=1
```

Production R2 upload should also require a second production allow flag after local/staging smoke evidence exists.

## Implementation Slices

### Slice 7B-4A: Transport Design Guardrail

- Add this document.
- Add static tests.
- Do not install R2 SDK.
- Do not parse `FormData` in the route yet.
- Do not upload.

### Slice 7B-4B: R2 Adapter Interface Only

- Add pure TypeScript adapter types.
- Add validation helpers.
- No SDK import.
- No route wiring.

### Slice 7B-4C: Route FormData Parser Model

- Add a pure parser/validator using test fixtures.
- Keep route disabled from R2 execution.
- No SDK import.

### Slice 7B-4D: Fake Adapter Route Test

- Dependency-inject fake metadata repository and fake R2 adapter.
- Verify order: claim -> image upload -> thumbnail upload -> finalize.
- Verify local payload deletion is only recommended after server success.
- Keep real R2 disabled.

### Slice 7B-4E: Real R2 Adapter

- Install the minimal AWS S3-compatible SDK only if required.
- Read only server-side R2 env variables.
- Add tests proving no public env or browser import.
- Do not enable production.

### Slice 7B-4F: Local/Staging Smoke

- Use disposable owner/staff test data.
- Upload one disposable image and thumbnail.
- Verify R2 private keys and Supabase uploaded metadata.
- Verify local payload deletion only after confirmed success.

### Slice 7B-4G: Production Decision

- Requires local/staging evidence.
- Requires manual production flag decision.
- Requires rollback/disable procedure.
- Requires monitoring of failed metadata rows and orphan candidates.

## Current Recommendation

Proceed next with `Slice 7B-4F: Local/Staging Smoke Wiring`.

Do not enable production upload or connect user-facing upload buttons until local/staging smoke evidence exists.
