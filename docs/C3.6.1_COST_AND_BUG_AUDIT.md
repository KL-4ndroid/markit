### 2.2 Egress 詳細計算

**單次 Polling 查詢的資料量**：
- Request headers: ~400 bytes
- Response (head:true, count): ~100 bytes
- **總計 ~500 bytes/查詢**

**Egress 計算（1000 / 10000 / 100000 / 1000000 在線員工）**：

| 員工數 | 60s Polling Egress/月 | 180s Polling Egress/月 | 差異 | Pro 250GB 配額影響 |
|---|---|---|---|---|
| 1000 | 720 MB | 240 MB | -480 MB | ✅ 都在 1% 內 |
| 10000 | **7.2 GB** | 2.4 GB | -4.8 GB | ✅ 都在 3% 內 |
| 100000 | **72 GB** | 24 GB | -48 GB | ✅ 都在 30% 內 |
| 1000000 | **720 GB** | 240 GB | -480 GB | ⚠️ 60s 超量 470GB × $0.09 = **$42/月** |

**結論**：
- ✅ **100000 員工以內**：60s 和 180s **都 0 額外成本**（在 Pro 250GB 配額內）
- ⚠️ **1000000 員工**：60s 開始計費，180s 仍在配額內
- 🟡 **Free 計畫**：10000 員工 60s 會吃 7.2GB（超 5GB 配額）→ $0.20/月超量費

### 2.3 DB CPU 負擔

**單次查詢成本**：
- 走 `(staff_id, status)` 索引
- 假設每次查詢 5ms（含 RTT）

**DB CPU 使用率**（單核 1000ms）：

| 員工數 | 60s Polling | 180s Polling |
|---|---|---|
| 1000 | 2.75% | 0.92% |
| 10000 | 27.5% | 9.2% |
| 100000 | **275%（>100% 需多核）** | 92% |

**等等** — 100000 員工 60s Polling **需要多核支援**。

**等等** — **等等** — 這是**單執行緒**估算（Postgres 查詢可平行）。**100000 員工 60s Polling 實際上可能仍是 OK**（Postgres 預設可以平行查詢）。

**Pro 計畫 compute 配額**：
- Micro: 2-core ARM, 1 GB RAM
- Small: 2-core ARM, 2 GB RAM
- Medium: 2-core ARM, 4 GB RAM

**等等** — 所有 Pro compute 都是 **2-core ARM**！**單核極限**！

**等等** — 但 Postgres 查詢可以在多個 connection 上平行。**多 connection = 多 query 平行**。

**100000 員工 60s Polling** = 1666 查詢/秒。每個查詢 5ms = **8.3 個 query 同時**。**2-core 處理 8.3 個 query 沒問題**。

**等等** — 但**單核 CPU 使用率**仍是 8.3 × 5ms / 1000ms = 4.15% 每核心。**總 CPU 使用 8.3%**。**沒問題**。

**等等** — 我**之前算錯了**！重新算：

**100000 員工 60s Polling**：
- 100000 員工 / 60 秒 = 1666 查詢/秒
- 每查詢 5ms
- **單核需要時間**：1666 × 5ms = **8330 ms CPU/秒** = **833% 單核** = **8.33 核**！

**等等** — **這超出 Micro/Small/Medium compute 的 2-core**！

**結論**：
- 100000 員工 60s Polling **需要 ≥ 8 cores**
- Pro Micro/Small/Medium (2-core) **不夠**
- **需要升級到 4XL (16-core) 或更高**

**180s Polling**：
- 100000 員工 / 180 = 555 查詢/秒
- 555 × 5ms = **2778 ms CPU/秒 = 2.78 核**
- **2-core 略不夠，需要 4-core**（Large 或 XL）

**等等** — 這**改變評估**！

### 3.3 重點分析：風險 #1（handleRevoked catch 路徑）

**現有程式碼（line 134-137）**：
```ts
} catch (error) {
  console.error('[StaffStatusMonitor] 清理失敗:', error);
  cleanupInFlightRef.current = false;  // ← 只 reset ref
}
```

**問題**：
- ❌ `window.location.href = '/'` 沒執行
- ❌ `indexedDB.deleteDatabase` 沒執行
- ❌ `sessionStorage.clear()` 沒執行
- ❌ 用戶卡在殘留狀態

**修法建議**：
```ts
} catch (error) {
  console.error('[StaffStatusMonitor] 清理失敗，強制重整:', error);
  cleanupInFlightRef.current = false;
  // ✅ 強制重整頁面（讓 React 重置 state）
  if (typeof window !== 'undefined') {
    window.location.href = '/';
  }
}
```

### 3.4 重點分析：風險 #12（多老闆 Polling 誤清空）

**現有 Polling 查詢**：
```ts
.eq('staff_id', user.id)
.eq('owner_id', userRole.ownerId!)  // ← 限定 owner
.eq('status', 'active')
```

**問題情境**：
1. 員工 B 同時是 A 和 C 的員工
2. B 登入後 `useUserRole` 回傳 `ownerId: A`（limit 1）
3. A 踢 B → `staff_relationships(owner=A, staff=B, status=active)` 被撤銷
4. **但** `staff_relationships(owner=C, staff=B, status=active)` **還在**（B 仍是 C 員工）
5. B 客戶端 Polling 查 `owner_id=A AND status=active` → `count=0` → **誤觸 handleRevoked**！
6. **B 仍是 C 員工但被清空所有資料**！

**修法建議**：
```ts
// ✅ 改為：查 B 是否仍是「任何老闆的員工」
const { count } = await supabase
  .from('staff_relationships')
  .select('*', { count: 'exact', head: true })
  .eq('staff_id', user.id)
  .eq('status', 'active');  // ← 不限定 owner_id

if (count === 0) {
  // B 不再是任何老闆的員工
  await handleRevoked('poll');
}
```

**等等** — 但這與 `userRole.ownerId` 的語意不同。

**等等** — 真正的修法應該是：
```ts
// 查 B 是否仍是 userRole.ownerId 這個特定老闆的員工
const { count } = await supabase
  .from('staff_relationships')
  .select('*', { count: 'exact', head: true })
  .eq('staff_id', user.id)
  .eq('owner_id', userRole.ownerId!)
  .eq('status', 'active');

if (count === 0) {
  // B 不再是 userRole.ownerId 這個老闆的員工
  // 但 B 可能在切換到老闆模式（userRole 可能也錯了）
  // 安全做法：呼叫 useUserRole 重新查詢，確認 isStaff 狀態
  await handleRevoked('poll');
}
```

**但如果 `useUserRole` 本身**用 `limit(1)` 錯誤判斷** isStaff**，那 Polling 沿用 `userRole.ownerId` **也是錯的**。

**根本修法**：
- 短期：保留現有 Polling 邏輯（接受多老闆情境的誤清空風險）
- 中期：修復 `useUserRole` 的 `limit(1)` 問題
- 長期：useUserRole 支援多老闆切換時，Polling 自動適應

**風險評估**：
- 🟡 **目前**：不會發生（useUserRole 不支援多老闆顯示，但 DB 支援）
- 🟡 **未來**：如果 useUserRole 改為支援多老闆，**會發生**

**等等** — 從 `lib/db/multi-account.ts` 看，**系統已經用 IndexedDB 切換多帳號**（不同 owner 不同的 DB）。所以**多老闆場景的「同一 supabase user 是多個 owner 的員工」是設計上的可能**。

**等等** — 但 `useUserRole` 的 `limit(1)` 會**在員工登入時只回傳第一個老闆**。**這個 bug 早就存在**，**不是 useStaffStatusMonitor 引入的**。

**建議**：**在 C3.6.2 修復 useUserRole 支援多老闆時，一併修復 Polling 邏輯**。

### 3.5 重點分析：風險 #17（closure trap）

**現有 Polling 邏輯**：
```ts
useEffect(() => {
  if (!userRole.ownerId) return;
  // ...
  const checkStaffStatus = async () => {
    // ← userRole.ownerId 在此 capture
    const { count } = await supabase
      .from('staff_relationships')
      .eq('owner_id', userRole.ownerId!)  // ← 使用 closure
  };
  // ...
}, [userRole.ownerId]);
```

**問題情境**：
1. `userRole.ownerId = A` → useEffect 建立
2. `userRole.ownerId` 變成 C → useEffect cleanup + 重新建立
3. **期間**：舊的 `checkStaffStatus` 還在跑（await 沒完成）
4. 舊的 closure 用 A 完成查詢 → 但 cleanup 已把 `isMounted = false`
5. 檢查 `isMounted === false` → return

**等等** — line 148 的 `if (!isMounted) return;` **是在 `await` 之後**。所以**await 完成後檢查 isMounted 會 return**。

**等等** — 但**等等** — handleRevoked 內的 async 邏輯（line 92-138）**沒有 isMounted 檢查**！

```ts
const handleRevoked = async () => {
  if (cleanupInFlightRef.current) return;
  cleanupInFlightRef.current = true;
  // ...
  await resetAuthenticatedCache('full');
  // ...
  window.location.href = '/';
};
```

**等等** — 如果 handleRevoked 觸發後，cleanup 函式被呼叫（依賴變了），但 handleRevoked 內部還沒執行到 `window.location.href`...

**等等** — `window.location.href = '/'` 是**同步**的，會立即中斷當前 JS。**所以** handleRevoked 內部**所有 await 之前的程式碼會執行**，**但** `window.location` 觸發後頁面跳轉，**新的 await 不會執行**。

**等等** — `window.location.href = '/'` 不是**真的同步**。瀏覽器處理 navigation 是**異步的**。但**對當前 JS 來說是同步的**（瀏覽器會繼續執行後續 JS，但**頁面準備卸載**）。

**等等** — 讓我仔細看：
- `await resetAuthenticatedCache('full')` - 等 IndexedDB 清理
- `await db.close()` - 等 DB 連接關閉
- `await new Promise(...)` - 等 IndexedDB deleteDatabase
- `sessionStorage.clear()` - 同步
- `resetInitialSyncFlag()` - 同步
- `window.location.href = '/'` - 觸發 navigation

**如果 useEffect cleanup 在第 2 個 await 之間觸發**（依賴變了）：
- `isMounted = false`（line 183）
- 清理 interval
- 新的 useEffect 跑（如果依賴變了）
- **但** `handleRevoked` 還在跑（await 中）

**情境 A**：handleRevoked 繼續完成 → `window.location.href` → **頁面跳轉**（沒問題）

**情境 B**：cleanup 設定了 `isMounted = false`，**新的 useEffect 跑時** 立即 polling（因為 `userRole.ownerId` 可能變了）：

**等等** — 新的 useEffect 用**新的 `userRole.ownerId`**，**邏輯上**應該 polling 新的 ownerId。**沒問題**。

**等等** — 但有個**競態**：
- 舊的 handleRevoked 用 A 的 ownerId polling → count=0 → 觸發 handleRevoked
- handleRevoked 開始清理
- **新的 useEffect 跑時**（依賴變了）→ 立即 polling → **可能 race**

**等等** — 但**新的 useEffect 不會立即 polling**。它的 `cleanupInFlightRef` 檢查會在 line 85-88 攔截：

```ts
if (cleanupInFlightRef.current) {
  console.log(`[StaffStatusMonitor] 清理已在進行中（${source}），跳過重複觸發`);
  return;
}
```

**等等** — 但 `cleanupInFlightRef` 是 **ref 物件**。**新 useEffect 建立的 handleRevoked closure 用同一個 ref**（line 69）：

```ts
const cleanupInFlightRef = useRef(false);
```

**等等** — `useRef` 在 component 生命週期內**是同一個物件**。**所以新舊 useEffect 共用同一個 ref**。

**新 useEffect 啟動時**：
- ref=true（因為舊 handleRevoked 設定的）
- 立即跑 checkStaffStatus → 用新 ownerId → 可能查到非 0 → **不會觸發 handleRevoked**（但 ref 已經是 true）

**OK 沒問題**。

**但等等** — 如果新 ownerId 查到的也是 0（**同樣被踢**）→ 觸發 handleRevoked → **但 ref 已經是 true** → return。

**OK 沒問題**。

### 3.6 其他重要風險

#### 風險 #6：catch 路徑強制 reload

**問題**：
```ts
} catch (error) {
  console.error('[StaffStatusMonitor] 清理失敗:', error);
  cleanupInFlightRef.current = false;
  // ❌ 沒執行 window.location.href
}
```

**建議修法**：
```ts
} catch (error) {
  console.error('[StaffStatusMonitor] 清理失敗，強制重整:', error);
  cleanupInFlightRef.current = false;
  // ✅ 強制重整（讓 React 重置 state，殘留資料不會被 useUserRole 誤讀）

  }
}
```

**但等等** — 強制 reload 後 useUserRole 重新查詢時，**雲端的 staff_relationships 已經是 revoked**（老闆已經踢了）。所以 useUserRole 會回傳 `isStaff: false`。**但 IndexedDB 還殘留**！

**等等** — 這就是問題：**reload 不會清空 IndexedDB**！

**修法**：
- 方案 A：catch 內繼續嘗試刪除 IndexedDB（即使前一步失敗）
- 方案 B：依賴 `useUserRole` 的 effect 偵測 isStaff 從 true 變 false → 自動清理
- 方案 C：使用者手動重整頁面（不理想）

**方案 B 最乾淨**，但需要修改 useUserRole（**超出 C3.6 範疇**）。

## 8. 決策記錄（Decision Log）

### 8.1 2026-06-15 最終決策：保留 180 秒 Polling 為預設

**決策**：
> ✅ **保留 180 秒 Polling 為預設值**

**決策理由**：

| 考量 | 60s | 180s |
|---|---|---|
| 現實規模成本（< 100 員工） | $0 | $0 |
| 100K 員工成本 | +$850/月 | 0 |
| 100K+ 員工 Egress | 需升級 | 在配額內 |
| 反應延遲 | 60 秒 | 180 秒 |
| **未來擴展性** | ❌ **100K 時需改回 180s** | ✅ **始終適用** |
| **程式碼變更次數** | ❌ **規模擴大需改 1 次** | ✅ **不用改** |
| **維護成本** | ❌ **需監控 + 動態調整** | ✅ **不變** |

**核心論點**：
- 對現實規模（< 100 員工）：60s 略優（反應快 3 倍，成本相同）
- 對未來規模（100K+ 員工）：180s 明顯優（省 $850/月）
- **規模擴大時如果預設是 60s，必須記得改回 180s**（**容易忘記**）
- **規模擴大時如果預設是 180s，不用改**（**不會出錯**）

**保守原則**：
> 「能不改的就不要改」—— 保留 180s 為預設，避免未來擴展時的隱性成本。
