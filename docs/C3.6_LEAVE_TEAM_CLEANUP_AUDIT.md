### 為什麼 v2.2 保留 180s 為預設？

**使用者決策**（2026-06-15）：
> 「保留 180s 為預設」

**理由**：
- ✅ 對現實規模（< 100 員工）：60s 和 180s 成本相同
- ✅ 對未來規模（100K+ 員工）：180s 節省 +$850/月 compute 升級費
- ✅ **規模擴大時不用改**（避免忘記改的隱性成本）
- ✅ 「能不改的就不要改」保守原則

**可選覆寫**：Hook 支援 `pollIntervalMs` 參數，未來如果需要可改為 60 秒或 Realtime。

### 2.2 情境 B：老闆踢人

**觸發路徑**：`components/settings/StaffManagement.tsx` → `handleRemove`（line 172-189）

**清理流程**：

```
[1] await removeStaff(staffId)  ← lib/supabase/staff.ts line 264-316
    ├── 查詢 staff_relationships (status IN pending/active)
    ├── revokeStaff(relId) → UPDATE staff_relationships.status = 'revoked'
    ├── 查詢老闆自己的 markets
    └── DELETE market_members WHERE user_id = staffId AND role = 'staff' AND market_id IN (老闆的市集)
        → 觸發 audit_logs (DELETE_role) ← 給 was_permission_valid_at 用的時間錨點

[2] 重新載入員工列表（loadStaffList）

[3] ❌ 沒有通知員工 B
[4] ❌ 沒有清空員工 B 的 IndexedDB
[5] ❌ 沒有清空員工 B 的 role cache
```

**評估**：🟡 **雲端清理完整，但客戶端沒有即時通知**

**問題清單**：
- ❌ **員工 B 不會知道被踢**（沒有 Realtime 推送）
- ❌ **員工 B 的 IndexedDB 殘留**（8 個自己的市集 + 從 A 拉的市集都還在）
- ❌ **員工 B 的 role cache 過期時間 5 分鐘**（`ROLE_CACHE_TTL_MS`，useUserRole.ts line 28）
- ❌ **員工 B 繼續看到「員工模式」**（因為 role cache 還在，isStaff 還是 true）
- ❌ **員工 B 繼續看到 A 的市集**（從 A 拉到的市集 + products + events 還在 IndexedDB）

**修法選項**：
1. **Supabase Realtime**：訂閱 `staff_relationships` 表的變更，被踢時自動清空
2. **TTL 縮短**：把 `ROLE_CACHE_TTL_MS` 從 5 分鐘縮短到 30 秒（折衷方案）
3. **Polling**：員工模式啟用時，每 30 秒查一次 `staff_relationships.status`
4. **廣播機制**：透過 BroadcastChannel 廣播（只對同瀏覽器的多分頁有效）

## 4. 老闆踢人（情境 B）的詳細問題

### 4.1 觸發鏈

```
老闆 A 在 StaffManagement UI 按下「移除員工」
    ↓
StaffManagement.tsx handleRemove(staffId, email)
    ↓
confirm() 確認
    ↓
await removeStaff(staffId)  ← lib/supabase/staff.ts
    ↓
[雲端清理] revoke + 清除 market_members
    ↓

    ↓
loadStaffList()  ← 重新載入老闆端的員工列表
    ↓
[END] ← 員工 B 端完全沒通知
```

### 4.2 員工 B 端的後果

**在 5 分鐘內**（role cache TTL）：
- `useUserRole()` 仍然回傳 `isStaff: true`
- `useSync` 走 `pullEventsFromViews` 員工邏輯
- B 繼續看到 A 的市集（殘留 + 從雲端拉的新資料）

**5 分鐘後**（cache 過期）：
- `useUserRole()` 重新查詢 → 發現 `staff_relationships.status = 'revoked'` → 回傳 `isStaff: false`
- `useSync` 切換到老闆邏輯
- 但 **B 自己的 8 個市集（owner 角色）會被重新拉取** — 因為 B 是 owner，RLS 允許 B 看到自己的市集
- **A 的市集**仍殘留在 B 的 IndexedDB（不會自動清空）

**等等** — 切換到老闆邏輯後，`pullAllEvents` 拉的是 B 自己的 events + 0d21abfe 的 21 個市集 events，**不包含 A 的市集 events**。**但 A 的市集資料本身**（markets、products 資料）**仍殘留在 IndexedDB**。

### 4.3 員工 B 端的 IndexedDB 殘留表

| 表 | 殘留狀況 | 是否會被自動清空 |
|---|---|---|
| `markets` | A 的市集資料 | ❌ **不會**（除非用戶主動清） |
| `products` | A 的市集內商品 | ❌ **不會** |
| `events` | A 的市集內 events | ❌ **不會**（但下次 sync 增量不會拉） |
| `dailyStats` | A 的市集內統計 | ❌ **不會** |

### 4.4 安全性評估

**等等** — 雖然 IndexedDB 殘留，**但**：
- 雲端 RLS 已經把 B 排除（A 的 events RLS 限縮到 actor_id = A 或 market_id IN A's markets）
- B 重新登入時，`useSync` 走老闆邏輯，**只拉自己的 events**（不含 A 的）
- **但 A 的市集資料殘留** — 這是 **local 視覺殘留**，**不是安全漏洞**
- 重新整理頁面後，`useLiveQuery` 從 IndexedDB 讀，仍會看到 A 的市集

**真正的問題**：
- 🟡 **資料隔離違規**（B 看到 A 的市集殘留）
- 🟡 **不是安全漏洞**（雲端 RLS 正常）
- 🟡 **但違反「Local-First 數據隔離」原則**

## 6. 「老闆踢人」議題的修法（**2026-06-15 已實作**）

### 6.1 修法 1：Supabase Realtime 訂閱（推薦）

```ts
// 員工端訂閱 staff_relationships 表的變更
const subscription = supabase
  .channel('staff-status')
  .on(
    'postgres_changes',
    { event: 'UPDATE', schema: 'public', table: 'staff_relationships' },
    (payload) => {
      if (payload.new.staff_id === currentUserId && payload.new.status === 'revoked') {
        // 被踢了！清空本地 + 重新整理
        await clearLocalAppData();
        window.location.href = '/';
      }
    }
  )
  .subscribe();
```

**優點**：
- ✅ 即時反應（秒級）
- ✅ 雲端單一事實來源
- ✅ 訂閱成本低

**缺點**：
- 🟡 需要 Supabase Realtime 已啟用（通常預設啟用）
- 🟡 連線斷掉時不會通知（需要 fallback polling）

### 6.2 修法 2：Polling 機制

```ts
// 員工模式啟用時，每 30 秒查一次
useEffect(() => {
  if (!isStaff) return;
  
  const checkStaffStatus = async () => {
    const { data } = await supabase
      .from('staff_relationships')
      .select('status')
      .eq('staff_id', user.id)
      .eq('status', 'active')
      .limit(1);
    
    if (!data || data.length === 0) {
      // 被踢了！清空 + 重新整理
      await clearLocalAppData();
      window.location.href = '/';
    }
  };
  
  const interval = setInterval(checkStaffStatus, 30000);
  return () => clearInterval(interval);
}, [isStaff, user]);
```

**優點**：
- ✅ 簡單實作
- ✅ 不依賴 Realtime
- ✅ 連線斷掉也會重連

**缺點**：
- 🟡 30 秒延遲（可以接受）
- 🟡 流量略增（每用戶每分鐘 2 次查詢）

## 7. 實作結果（2026-06-15）

**結合修法 1（Realtime） + 修法 2（Polling 備援）**已實作於 `hooks/useStaffStatusMonitor.ts`：

```ts
// 完整實作（已上線）
useEffect(() => {
  if (!enabled || !user || !isStaff || !userRole.ownerId) return;

  // [主要] Realtime 訂閱 staff_relationships
  const channel = supabase
    .channel(`staff-status-monitor-${user.id}`)
    .on('postgres_changes', {
      event: 'UPDATE', schema: 'public', table: 'staff_relationships',
      filter: `staff_id=eq.${user.id}`,
    }, (payload) => {
      const newRow = payload.new as { status?: string };
      const oldRow = payload.old as { status?: string };
      if (newRow.status === 'revoked' && oldRow.status === 'active') {
        void handleRevoked('realtime');
      }
    })
    .subscribe();

  // [備援] 60 秒 Polling
  pollInterval = setInterval(() => { void checkStaffStatus(); }, 60_000);
  void checkStaffStatus();  // 立即跑一次

  return () => {
    isMounted = false;
    if (pollInterval) clearInterval(pollInterval);
    void supabase.removeChannel(channel);
  };
}, [enabled, user, isStaff, userRole.ownerId]);
```

**評估**：
- ✅ Realtime 處理即時（秒級）
- ✅ Polling 處理連線斷掉

### 9.2 Postgres Changes 的額外負擔

> 來源：[Supabase Postgres Changes](https://supabase.com/docs/guides/realtime/postgres-changes)

1. **邏輯複製**：
   > "Postgres Changes use logical replication to stream database changes from the Write-Ahead Log (WAL) to subscribed clients."

2. **單執行緒瓶頸**：
   > "Postgres Changes process changes on a single thread to maintain order, which can create bottlenecks at scale compared to Broadcast."

3. **N 次 RLS 檢查**：
   > "For the Postgres Changes feature, every change event must be checked to see if the subscribed user has access. For instance, if you have 100 users subscribed to a table where you make a single insert, it will then trigger 100 'reads': one for each user."

4. **官方建議改用 Broadcast**：
   > "Postgres Changes are simple to use, but have some limitations as your application scales. We recommend using **Broadcast** for most use cases."

### 9.3 `supabase_realtime` publication 預設未啟用

- 預設 `supabase_realtime` publication 為空
- 需要手動 `ALTER PUBLICATION supabase_realtime ADD TABLE staff_relationships`
- **目前專案 migration 沒有啟用** → v1 Realtime 訂閱在生產環境會失敗

### 9.4 三個方案成本對比（10000 在線員工）

| 方案 | 連線 | 訊息 | 邏輯複製 | DB 查詢 | 月成本 |
|---|---|---|---|---|---|
| v1 Realtime + 60s Polling | 10000 峰值 | ~50000 | 高 | 4.3M | **$100+** |
| **v2 Polling 180s + head:true** | 0 | 0 | 無 | 1.44M | **$0** |
| Polling 300s（等 useUserRole cache 過期） | 0 | 0 | 無 | 0 | $0（需修 useUserRole） |

### 9.5 v2 選擇的權衡

**接受 180 秒延遲**換取：
- ✅ **節省 $100+/月**（10000 員工規模）
- ✅ **零 Realtime 連線**
- ✅ **零邏輯複製壓力**
- ✅ **零 RLS 過濾負擔**
- ✅ **不需要啟用 publication**

**180 秒延遲在哪些情境可接受**：
- ✅ 老闆踢人後，員工 B 在 3 分鐘內自動清空
- ✅ 員工 B 不在線時（背景分頁）→ 不影響
- ✅ 員工 B 重新整理頁面時 → 不影響（userRole hook 重新查詢）
- 🟡 員工 B 正在看「員工管理」頁面時 → 需等 3 分鐘（但 B 不會看到自己的管理頁面，B 是被管理對象）

### 9.6 Polling 查詢效能分析

**查詢語句**：
```sql
SELECT count(*) FROM staff_relationships
WHERE staff_id = ? AND owner_id = ? AND status = 'active'
LIMIT 1
```

**索引覆蓋**：
- `idx_staff_relationships_staff` ON `(staff_id, status)`（從 migration）
- 直接走索引，O(log n) lookup

**10000 員工 × 1/180s = 55 查詢/秒**：
- 假設每次查詢 5ms
- 55 × 5ms = 275ms/秒 CPU
- 單核 1000ms → **27.5% DB CPU**

**比 Realtime 邏輯複製還輕**！

### 9.7 為什麼不用 Broadcast？

**Broadcast 機制**：
- 老闆踢人 → `removeStaff()` 完成後主動發 broadcast
- 員工端訂閱 broadcast 事件

**缺點**：
- 🟡 需要修改 `removeStaff`（增加 broadcast 邏輯）
- 🟡 需要在 supabase 端接收 broadcast（trigger 或 RPC）
- 🟡 跨瀏覽器/裝置依賴 realtime connection
- 🟡 還是會消耗 Realtime 連線

**vs Polling**：
- Polling 只需要 hook，**零額外基礎設施**
- Polling 不需要觸發端配合
- 180 秒延遲可接受 → **Polling 更簡單**

### 9.8 為什麼不用 `useUserRole` cache 過期機制（300 秒）？

**理論方案**：
- 不加 hook
- 讓 `useUserRole` cache 在 300 秒後過期
- 過期時 `useUserRole` 重新查詢 → 發現 `isStaff: false` → 設 state
- 在 `useUserRole` 內加 effect：當 `isStaff` 從 true 變 false 時清空 IndexedDB

**問題**：
- 🟡 300 秒延遲（比 180 秒慢 120 秒）
- 🟡 需要修改 `useUserRole.ts`（**現有 useUserRole 5 分鐘 cache 已設計為「容忍短暫不一致」**，不應被清空邏輯污染）
- 🟡 `useUserRole` 的職責應該專注於「查角色」，不應包含「清理 IndexedDB」副作用

**v2 折衷**：
- ✅ 獨立 hook，職責單一
- ✅ 180 秒延遲（比 cache 過期快 2 分鐘）
- ✅ 不污染 `useUserRole`
