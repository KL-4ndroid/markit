# C3.5 老闆接受員工邀請後資料殘留問題（議題記錄）

> **建立日期**：2026-06-14
> **建立者**：Cursor (Codex)
> **狀態**：📋 議題記錄，待 root cause 確認
> **嚴重度**：🟡 P1（功能性，非安全性）

---

## 1. 問題描述

### 1.1 使用者回報

**情境**：
- 使用者建立「員工 B」帳號（`85d316ce-5871-4311-ade3-c11b766b69ab`）
- 員工 B 本身也是 supabase 的**獨立老闆身份**（擁有 8 個市集、~500+ 個 events）
- 員工 B 使用「老闆 A」發出的邀請連結登入
- 在按下「接受邀請」**之前**，B 仍然以「老闆」身份運作（`useUserRole` fail-open → `isStaff: false`）
- 員工 B 按下「接受邀請」後，`useUserRole` 變 `isStaff: true` → 走 `pullEventsFromViews` 員工邏輯

**疑慮**：
> 「老闆帳號似乎會載入所有的資料庫資料」

---

## 2. 完整診斷結果（2026-06-14 SQL 驗證）

### 2.1 RLS 過濾數字

| User | 透過 RLS 可看到的 events | 佔總 events |
|---|---|---|
| A (`35c253c0-...`) | **236** | 6.7% |
| B (`85d316ce-...`) | **672** | 19.2% |
| 不可見（其他人） | 2601 | 74.1% |

**結論**：
- ✅ **RLS 真的有在過濾**（A 跟 B 都只看到部分）
- ✅ **沒有「看到所有人」的安全漏洞**

### 2.2 A 的市集成員

A 只有 3 個市集（test1, test11, test5），全部 owner = A，**沒有被其他老闆邀請的記錄**。

### 2.3 B 的市集成員（**驚人發現**）

B 同時服務於**多個老闆**：

| 角色 | 數量 | owner_id | 說明 |
|---|---|---|---|
| owner | 8 個 | 85d316ce-... (B 自己) | B 是老闆 |
| staff | 3 個 | 35c253c0-... (A) | B 是 A 的員工 |
| owner | 21 個 | **0d21abfe-... (第三人)** | **B 是第三人的 owner**（21 個市集全部 owner_id = 0d21abfe） |

**等等** — 這 21 個市集是「B 在 0d21abfe 底下做事」時的市集，但 `market_members.role = owner` 表示 B 是 owner 角色。

**等等** — **0d21abfe-... 是誰？** 可能是 supabase 第三個 user，也可能是「員工 B 的老闆」（B 為 0d21abfe 工作）。

### 2.4 staff_relationships（**A 跟 B 互為對方**）

| 關係 | Status | 意義 |
|---|---|---|
| `35c253c0 (A) → 85d316ce (B)` | **revoked** | **A 曾是 B 的員工**（已撤銷） |
| `85d316ce (B) → 35c253c0 (A)` | **active** | B 是 A 的員工（啟用中） |

**重要發現**：**A 跟 B 互為對方老闆/員工**（曾經），現在 B 仍是 A 的員工，但 A 已被 B revoke。

### 2.5 C2.27 revoke 清理邏輯（時間窗口設計）

**`was_permission_valid_at(UUID, TIMESTAMPTZ)` 函式**（20240220_add_staff_roles.sql line 460-494）：

```sql
-- 查詢邏輯：
-- 1. 找 market_members 中 role='staff' 的最後一次建立時間
-- 2. 從 audit_logs 找對應的 DELETE_role 時間
-- 3. 判斷 [created_at, deleted_at) 時間窗口
```

**設計重點**：
- 不是用 `staff_relationships.status` 直接決定權限
- 而是用 **時間窗口**（grant 開始、revoke 結束）
- 每個事件查自己的 timestamp 是否在權限有效期內

**市場級的 events** 不透過這個函式過濾（市場是「時空固定」），但**其他類型 events** 會。

### 2.6 A 跟 B 互為對方的特殊情境

**A 曾是 B 的員工**（revoked）：
- A 為 B 工作時，可能在 B 的市集內有 events（`actor_id = A, market_id IN B's markets`）
- 這些 events 在 A_sees 中**可能仍然被 RLS 看到**（市場未清空記錄）
- 但 `was_permission_valid_at` 透過時間窗口過濾「A 寫入時」的權限

**結論**：
- A_sees = 236 是「被時間窗口過濾後」的數字
- A 為 B 工作時的 events 透過 `audit_logs` 的 `DELETE_role` 時間點判斷
- 這是 C2 議題的「**最小權限時間窗口**」設計

## 3. 「老闆帳號載入所有資料庫資料」議題的真相

### 3.1 使用者指的可能對象

**「老闆帳號載入所有資料庫資料」**可能指以下其中之一：

1. **A 老闆** — A_sees = 236 個（合理範圍）
2. **B 員工（本身是老闆）** — B_sees = 672 個（含 B 自己的 8 個市集 + A 的 3 個市集 + 0d21abfe 的 21 個市集）
3. **0d21abfe 老闆** — 還沒查，應該是 21 個市集的所有 events

### 3.2 如果指 A：

- A_sees = 236 = A 自己的 3 個市集 + 106 個 null_market_id + 部分時間窗口內的 events
- **沒有「載入所有」** — 看到的是「A 有權限看的」
- ❌ 不是漏洞

### 3.3 如果指 B（員工 B 同時是老闆）：

- B_sees = 672 = B 自己的 8 個市集 + A 的 3 個市集 + 0d21abfe 的 21 個市集 + 106 個 null_market_id
- B 透過 RLS 看到所有「B 服務的市集」的 events
- **這是 RLS 正確的行為**（B 同時是多個老闆的 owner/staff）
- ❌ 不是漏洞

### 3.4 真正的「疑慮」可能是**UI 顯示問題**：

**B 接受邀請後**：
- `useUserRole` 變 `isStaff: true`（因為 A→B 是 active）
- `infoLevel < 3` → 走 `pullEventsFromViews` 員工邏輯（C3.4 新邏輯）
- B 透過 `staff_accessible_events` 視圖看到 A 分享的 events
- **但 B 自己的 8 個市集（owner 角色）不該在員工 view 中**

**疑慮**：
> 「B 接受 A 邀請後，是否會看到 B 自己的 8 個市集？」

**答案**：
- ❌ 員工 view **不包含** B 自己的市集（B 是 owner 不是 staff）
- ✅ 但 B 的 IndexedDB **殘留**老闆階段的 8 個市集
- 🟡 **local 殘留 + view 合併 = 顯示「8 + 3 = 11 個市集」**

## 4. 真正的 Root Cause（**最終答案**）

### 4.1 假設重新評估

| 假設 | 驗證結果 |
|---|---|
| A. 587 = B 自己的 events | ❌ B_sees = 672（不是 587），A_sees = 236 |
| B. 接受邀請後 local 殘留 | ✅ **最可能**（B 自己是 owner + 員工） |
| C. A 載入 B 團隊 events | ❌ A_sees = 236 不是 672（沒拉到 B 的 8 個市集） |

### 4.2 Root Cause（**最可能**）

**B 用戶的特殊身份**：
- B 自己是 supabase 的**獨立老闆**（8 個市集、500+ events）
- B 同時是 A 的員工（3 個市集）
- B 還涉及第三人 `0d21abfe-...`（21 個市集）

**B 接受 A 邀請後**：
- `useUserRole` 變 `isStaff: true`（因為 A 對 B 是 active）
- 走 `pullEventsFromViews` 員工邏輯
- 透過視圖看到 A 分享的 events（**不含 B 自己的 8 個市集**）
- **但 IndexedDB 殘留老闆階段的 8 個市集 + 500+ events**
- **UI 同時顯示「員工視角（A 分享的 3 個市集）」+「老闆視角（B 自己的 8 個市集）」**

**這就是「老闆載入所有資料庫資料」的真相** — **B 接受邀請後，看到自己老闆時的資料 + 員工時的資料**。

### 4.3 不是漏洞的原因

- ❌ **沒有 RLS 漏洞**（A_sees=236, B_sees=672, 都被限縮）
- ❌ **沒有「看到其他人資料」**（B 只看到自己服務的市集）
- 🟡 **是 IndexedDB local 殘留**（B 切換角色時沒清空老闆階段的資料）

## 7. 完整的議題分析（最終版）

### 7.1 「老闆帳號載入所有資料庫資料」的真實含義

**經 SQL 驗證後的真實故事**：

1. **B 用戶的特殊身份**：
   - B 自己是 supabase 的**獨立老闆**（8 個市集、500+ events）
   - B 同時是 A 的員工（active）
   - B 還有 `0d21abfe-...` 第三人的 21 個市集 owner 記錄

2. **A 跟 B 互為對方的歷史**：
   - A 曾是 B 的員工（revoked）
   - B 是 A 的員工（active）
   - 這個「互為老闆」的設計可能導致**market_members 表混亂**

3. **B_sees = 672 個 events**：
   - 透過 RLS 看到 32 個市集的所有 events
   - 包含 B 自己的 8 個 + A 的 3 個 + 0d21abfe 的 21 個
   - **這是 RLS 正確的行為**（B 確實是這些市集的老闆/成員）

### 7.2 沒有 RLS 安全漏洞

- ✅ A_sees = 236（被限縮）
- ✅ B_sees = 672（被限縮）
- ✅ 不可見 = 2601（74.1% 被過濾）
- ✅ `was_permission_valid_at` 時間窗口設計有效

### 7.3 真正的「疑慮」可能不是「老闆載入」

**真正的議題可能是**：
- B 接受 A 邀請後，UI 同時顯示「老闆時的 8 個市集」+「員工時的 3 個市集」
- 這是 **IndexedDB local 殘留** 的問題
- 不是雲端 RLS 的問題

**建議修法**（**不動，僅記錄**）：
- 角色切換時清空 IndexedDB 殘留
- 或在 `useLiveQuery` 中過濾「不該看到的市集」

### 7.4 待釐清的第三人 `0d21abfe-...`

**B 的 21 個市集 owner = 0d21abfe-...，但 B 在這些市集的角色是 `owner`（不是 staff）**：
- 這違反一般設計邏輯（市集 owner 應該 = market.owner_id）
- 唯一可能：`0d21abfe-...` 是 supabase 第三個 user
- 透過某些機制將這些市集分享給 B 並標記為 owner

**建議**：跑 SQL 確認 `0d21abfe-...` 是誰，確認 `markets.owner_id` 跟 `market_members.user_id` 的關係
