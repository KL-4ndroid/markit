# 互動功能重新設計文檔

## 📋 設計理念

### 核心原則
**使用者不是在「設定功能」，而是在「說明他怎麼賣東西」**

這個原則貫穿整個設計，確保使用者體驗自然、直覺，不會迷失在技術術語中。

---

## 🎯 設計目標

### 避免的問題
❌ 出現「漏斗、轉換率、角色」等技術詞彙  
❌ 一開始就要求使用者輸入文字  
❌ 讓使用者面對複雜的設定選項  

### 追求的體驗
✅ 先用「選擇情境」引導  
✅ 再用「確認理解」建立信心  
✅ 最後才「允許自訂」提供彈性  

---

## 🚀 設定流程設計

### Step 0：為什麼要設定？
**目標：一句話說明價值**

```
記錄顧客互動，讓你知道哪一場市集效果最好
不需要精準，只要直覺點擊
```

- 不使用彈窗，直接在頁面上呈現
- 只有一個按鈕：【開始設定】
- 清晰傳達價值，降低心理門檻

---

### Step 1：你賣的是什麼？
**目標：用情境而非角色分類**

提供 5 種攤位類型選項：
- 🍔 食品 / 飲料
- 💍 飾品 / 配件
- 🎨 插畫 / 創作
- 👕 服飾
- 📦 其他

**設計要點：**
- 使用大圖示 + 清晰文字
- 一鍵選擇，無需解釋
- 基於「賣什麼」而非「你是誰」

---

### Step 2：我們幫你準備好了
**目標：直接給完成品，不先給設定**

展示預設的三個按鈕組合，例如（食品類）：
```
👋 試吃
💬 詢問
💰 購買
```

下方說明：
```
記錄顧客從「有興趣」到「成交」的過程
```

**兩個選項：**
- ✅【就用這組】（主按鈕）
- ✏️【我想調整】（次要按鈕）

**設計要點：**
- 預設組必須是主要選項
- 降低決策負擔
- 大多數使用者可以直接完成

---

### Step 3：三個按鈕的意義
**目標：在調整前建立正確理解**

⚠️ **只在使用者點「我想調整」時才出現**

使用三張卡片說明：

#### 卡片 1：有興趣
```
標題：第一個：有興趣
說明：顧客停下來看、拿起、試試看
      代表他對你的商品有興趣
範例：試吃 / 拿起 / 翻看
```

#### 卡片 2：有互動
```
標題：第二個：有互動
說明：顧客開始跟你說話、問問題
      代表他有購買或了解的意圖
範例：詢問價格 / 聊材質 / 問來源
```

#### 卡片 3：轉換
```
標題：第三個：轉換
說明：顧客完成你想要的行為
      不一定是買，但對你有價值
範例：購買 / 加 IG / 加 Line / 訂單
```

**底部定心丸：**
```
只要照這個順序設定，後面的分析才會準
```

**設計要點：**
- 用「說人話」的方式解釋
- 不使用長文，用卡片分隔
- 強調順序的重要性
- 建立使用者信心

---

### Step 4：自訂畫面
**目標：極度克制的自訂選項**

固定三列，不能增減：
```
[ 有興趣 ]   顯示名稱：____   Icon：👋
[ 有互動 ]   顯示名稱：____   Icon：💬
[ 轉   換 ]   顯示名稱：____   Icon：💰
```

每列下方的提示（淡灰色）：
- 有興趣：顧客注意到你的商品
- 有互動：顧客跟你有對話
- 轉換：顧客完成你想要的行為

**限制：**
❌ 不提供新增  
❌ 不提供刪除  
❌ 不讓調整順序  

**設計要點：**
- 保持三階段漏斗結構穩定
- 只允許修改顯示文字和圖示
- 確保數據分析的一致性

## 💡 為什麼這樣設計一定會成功？

### 對使用者來說
1. **降低認知負擔**
   - 只需「選擇 + 確認」
   - 沒有被丟進設定地獄
   - 感覺「系統懂我怎麼賣」

2. **建立信心**
   - 先看到完成品
   - 理解後再調整
   - 每一步都有明確指引

3. **直覺操作**
   - 基於真實場景
   - 使用日常語言
   - 沒有技術術語

### 對系統來說
1. **數據穩定性**
   - 三個角色永遠固定
   - 所有分析都算得起來
   - 不會因自訂而破壞結構

2. **可擴展性**
   - 未來可加 AI 分析
   - 可加指標比較
   - 可加轉換率計算
   - 數據結構不會炸

3. **維護性**
   - 邏輯清晰
   - 限制明確
   - 不會產生邊界情況

## 🏗️ 技術實作

### 數據結構
```typescript
interface InteractionButton {
  id: string;           // 固定：'interest', 'engage', 'convert'
  role: 'interest' | 'engage' | 'convert';  // 角色（內部使用）
  label: string;        // 顯示名稱（使用者可自訂）
  emoji: string;        // 圖示
  description: string;  // 說明文字（內部使用）
}
```

### 核心原則
- `id` 和 `role` 永遠固定，確保數據一致性
- `label` 和 `emoji` 可自訂，提供個性化
- `description` 用於引導，不暴露給最終使用者

### 儲存策略
1. **本地優先**：localStorage 快速存取
2. **雲端同步**：登入後自動同步到 Supabase
3. **事件驅動**：使用 storage 事件通知其他頁面更新

## 📝 設計心得

這次重新設計的關鍵在於：

1. **同理心**：站在攤主的角度思考，他們關心的是「怎麼賣」而非「怎麼設定」

2. **漸進式揭露**：不一次給太多選項，讓使用者逐步建立理解和信心

3. **限制即自由**：通過限制選項（固定三個按鈕、固定順序），反而讓使用者更容易做決定

4. **說人話**：避免技術術語，用日常語言描述功能

5. **完成品優先**：先展示結果，再允許調整，降低決策焦慮

這些原則不僅適用於這個功能，也可以應用到整個產品的其他部分。
