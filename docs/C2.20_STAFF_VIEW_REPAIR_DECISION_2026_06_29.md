# C2.20 Staff View Repair Decision

Date: 2026-06-29

Status: decision required. This document does not approve or apply a migration.

## Evidence

C2.20 read-only verification found three deployed staff view/sanitizer issues.

### 1. `staff_accessible_markets` owner-branch scope leak

The deployed view still has an owner branch shaped like:

```sql
FROM markets m
JOIN market_members mm ON mm.market_id = m.id
WHERE mm.user_id = auth.uid()
```

Because staff accounts are inserted into `market_members`, a staff session can hit this owner branch and receive full owner fields.

Observed result:

```text
total_markets: 87
booth_cost_null: 46
total_profit_null: 46
commission_rate_null: 59
registration_fee_null: 46
```

Expected staff behavior: sensitive-column null counts should equal `total_markets`.

### 2. `staff_accessible_events` duplicate/sanitizer bypass path

The deployed events view has staff branches that sanitize payloads, but also has:

```sql
WHERE e.actor_id = auth.uid()
```

For staff-authored market events, this can return the same event once through the staff branch and once through the owner-style self branch.

Observed result:

```text
duplicate event id rows: 9
highest duplicate_count: 2
types: deal_closed, deal_deleted, interaction_deleted, interaction_recorded
```

Risk:

- duplicate local replay work;
- possible unsanitized payload path for staff-authored events.

### 3. `deposit` is not removed by cloud sanitizer

`sanitize_staff_event_payload` does not remove `deposit`.

C2.20-I confirmed staff-visible `market_created` payloads include `deposit`, including non-zero values.

Local code treats `deposit` as sensitive for staff:

- `lib/permissions/PermissionGate.ts` includes `deposit` in market/event sensitive fields.
- `tests/data-sanitization.test.ts` asserts staff market `deposit` must be removed.

## Why This Is High-Risk

Repairing this requires changing Supabase views/functions used by staff sync. That can affect:

- staff market hydration;
- staff event replay;
- tombstone visibility;
- owner view behavior if branches are changed incorrectly;
- future C2.29B verification.

No automatic repair should be applied without approval.

## Options

### Option A - Recommended: focused repair migration

Draft a new migration that:

1. Updates `sanitize_staff_event_payload` to remove `deposit`.
2. Rebuilds `staff_accessible_markets` so owner branch uses strict ownership, not `market_members`.
3. Rebuilds `staff_accessible_events` so staff sessions do not hit owner-style unsanitized branches.
4. Preserves tombstone visibility for staff.
5. Preserves owner full access.
6. Adds read-only verification blocks for:
   - staff market sensitive columns all null;
   - staff event payload sensitive keys all zero, including `deposit`;
   - staff event duplicate ids zero;
   - owner branch still works.

Benefits:

- Fixes all confirmed C2.20 issues in one coherent view/sanitizer repair.
- Aligns cloud sanitizer with local PermissionGate.
- Reduces duplicate replay risk before C2.29B.

Risks:

- Changes staff sync source views.
- Requires careful Supabase manual execution and post-apply verification.

### Option B - Minimal sanitizer-only repair

Only update `sanitize_staff_event_payload` to remove `deposit`.

Benefits:

- Smaller scope.

Risks:

- Does not fix `staff_accessible_markets` owner-branch leakage.
- Does not fix duplicate staff events.
- C2.20 would still fail.

### Option C - Scope-only repair, leave `deposit`

Fix view branch leakage and duplicate event ids, but keep `deposit` visible.

Benefits:

- Avoids changing the older product decision that treated deposit as an operational reminder.

Risks:

- Conflicts with current local PermissionGate and tests.
- Leaves a known sensitive-field inconsistency.

## Recommendation

Use Option A, but as a separate approved migration draft step.

Do not proceed to C2.29B until C2.20 passes after repair.
