## 5. 修復後的 invariant（不變式）

**這些是 C3.4 修好後應永遠成立的特性。若發現任何一條不成立 → bug 已回歸。**

### 5.1 UI 內部一致性

> 詳情頁下方總計 = 每日卡片加總 = `MarketCard` 顯示的總計

**驗證**（手動）：
- 進水水市集詳情頁
- 看到下方總計應與列表卡片「$100,376 / 6 筆」一致
- 加總每日卡片 = 同一數字

### 5.2 同步後不污染

> Owner 同步 N 次（無新 events）後，本地 `market.totalRevenue` 不變

**驗證**（console）：
```js
const m = await db.markets.get('<market-id>');
console.log('revenue:', m.totalRevenue, 'deals:', m.totalDeals);
// 連續按 3 次「立即同步」後，這兩個數字不應變
```

### 5.3 員工端永遠 observation-only

> Staff view 永遠不會 destructive rebuild projection

**驗證**（自動）：跑 §2.3 測試中的 `staff-view + inflated does NOT trigger repair`

### 5.4 useMarketStatsBatch 跨市集隔離

> `useMarketStatsBatch([A, B])` 中 `batchStats[A].totalRevenue` 不含 B 的 events

**驗證**（自動）：跑 §2.4 測試

### 5.5 Stock 仍從雲端帶入（故意）

> `product.stock` 從雲端同步後是雲端值（不是 0）

**驗證**（手動）：
```js
const p = await db.products.get('<product-id>');
console.log('stock:', p.stock, 'totalSold:', p.totalSold);
// stock 應是雲端的絕對剩餘量
// totalSold 應是 0（會被後續 events replay 累加）
```

### 6.5 Staff view 的污染處理（不是 bug）

Staff view 的 pollution **不**是 C3.4 的 bug，而是 P2 故意設計成 observation-only。

**員工端污染的處理路徑**：

1. 員工不要自己進 Recovery 頁（員工端 partial events）
2. 由 owner 端登入 → 觸發 owner-full sync → P2 自動 rebuild → 雲端正確
3. 員工下次登入會拉到雲端正確值

或若 owner 端也沒 rebuild（例如雲端本身就錯）：見 §7.3。

## 7. 已知 P4+ 議題（C3.4 無法解，列為後續）

### 7.1 Product.stock 雙重扣減（**P4 候選**）

**症狀**：`deal_closed` replay 後 `product.stock` 變負數或拋「庫存不足」錯誤

**為什麼 C3.4 沒修**：
- stock 是「絕對剩餘量」，從雲端帶入是合理的
- 但若雲端 stock 已被雲端 handler 扣過，本地 replay 又扣一次
- handler 有 `Math.max(0, ...)` 保護（L833-836），**但這在 P1 reset 之前已污染的 stock 上仍會誤算**

**相關程式碼**：`lib/db/events.ts:832-839`

**未來修法**（見 `PROJECTION_DOUBLECOUNT_FIX_PLAN.md §九.1`）：
- 方案 A：staff view 不扣 stock，只更新 totalSold
- 方案 B：比對 events list 完整性，缺的不 replay
- 方案 C：在 db.products 寫入時不帶 stock，由本地 handler 從 0 開始維護

### 7.2 MarketDetail line 1712 / 1559 抽成與淨利潤（**P5 候選**）

**症狀**：`MarketDetail` 的「抽成」/「淨利潤」仍讀 `market.totalRevenue` / `market.totalProfit`，即使 C3.4 修好 sync，**若用戶手動污染** `db.markets.totalRevenue` 仍會錯

**為什麼 C3.4 沒修**：
- P0 改了 `DailyRevenueStats` 下方總計（最常見的 UI 矛盾）
- `MarketDetail` 內部抽成 / 淨利潤是次要 UI，不是水水市集截圖的矛盾源

**未來修法**：
- 抽成 / 淨利潤也改用 `dailyData.reduce()`
- 或在 `useLiveQuery` 層做一道「讀 `dailyStats`，fallback 到 `market.total*`」

### 7.3 雲端本身就污染（C3.4 不改雲端）

**症狀**：
- 雲端 `markets.total_revenue` ≠ events 累加
- 本地即使 P0+P1+P2+P3 都修好，仍拉到雲端污染值

**處理路徑**：

1. **診斷**：跑 `docs/CLOUD_DATA_CONSISTENCY_AUDIT.md` A 查詢
2. **修雲端**：手動 SQL 從 events 重新計算並 update `markets.total_revenue`
3. **修本地**：P2 auto-repair 會在下次 owner sync 時用 events 重建

### 7.4 Snapshot 路徑污染（**observation-only**）

**症狀**：snapshot 載入後 `db.markets` 被覆蓋，可能帶入雲端污染

**為什麼 P2 不修**：
- Snapshot 邏輯複雜、覆蓋直接
- Recovery 頁是更好的修復點
- 已被 P2 `shouldAutoRepairForContext('snapshot') = false` 守住

**未來修法**：由 Recovery 頁面「重建 snapshot 來源」按鈕手動引導。

### 7.5 Status === 'lower_than_events'（**P2 故意不修**）

**症狀**：雲端 projection 比 events 累加少

**為什麼 P2 不修**：
- 雲端可能因為 RLS / view 過濾了部分 events
- 「雲端少 = 正確」是合理情境（員工看不到的成本不應顯示）
- 第一版保守，避免誤刪

**未來修法**：若用戶回報「該修沒修」，加 P2.5 偵測「雲端 0、本地 > 0 且 events 累加 = 本地值」時觸發 rebuild。

## 8. 升級 / 部署檢查清單

C3.4 對**老用戶**的升級效果：

1. **第一次 sync 之前**：本地 `db.markets.totalRevenue` 可能是污染值
2. **第一次 sync 時**：
   - P1 把雲端 hydrate 的 `totalRevenue = 0` 寫入
   - P1 之後 replay 雲端 events → 正確累加值
   - P2 偵測到 `status === 'inflated'`（既有污染 vs 雲端 events 算出值）→ auto-rebuild
3. **第二次 sync 起**：P0+P1+P2 都正確，無污染

**預期行為**：升級後老用戶第一次同步會看到數字從「污染值」跳到「正確值」（這是預期，不是 bug）。

**release notes 應包含**：

> **C3.4 Projection 同步修復**：詳情頁總計現在會與列表卡片自動對齊。升級後第一次同步可能會看到數字更新到正確值（過去的本地污染會自動被雲端 events 重建）。
