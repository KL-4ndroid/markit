## 3. C2.29B-2.1：收緊底表 SELECT RLS

### 3.1 目標

讓 `markets` / `products` / `events` 的 SELECT 政策**對員工 vs Owner 有不同行為**：

| 表 | Owner SELECT | Staff SELECT |
|---|---|---|
| `markets` | 完整欄位 | **拒絕**（員工只能走 view）|
| `products` | 完整欄位 | **拒絕**（員工只能走 view）|
| `events` | 完整欄位 | **拒絕**（員工只能走 view）|

### 3.2 核心策略：**Owner 判斷式用 `m.owner_id` / `p.owner_id` / `e.actor_id`**

採**未來好維護**的策略 — **不依賴 `market_members` 觸發器**（避免 021 trigger 漏加 / 002 trigger 漏加 owner row 的風險）。

#### 3.2.1 `markets` SELECT 政策

**現狀**（035 L195-200）：

```sql
CREATE POLICY "markets_select_secure"
ON public.markets FOR SELECT
TO authenticated
USING (
  id IN (SELECT * FROM public.current_user_market_ids())
);
```

**C2.29B-2.1 規劃**：

```sql
DROP POLICY "markets_select_secure" ON public.markets;

-- Owner branch: 直接從 markets.owner_id 判斷
CREATE POLICY "markets_select_owner_only"
ON public.markets FOR SELECT
TO authenticated
USING (owner_id = auth.uid());

-- Staff branch: 拒絕（員工必須走 view）
-- （無 policy = 自動拒絕）
```

**為何這樣改**：

- ✅ **不依賴 `market_members`**：直接從事實欄位 `owner_id` 判斷
- ✅ **不依賴 002 trigger**（market_created 自動加 owner 到 market_members）
- ✅ **Owner 可 SELECT 自己所有市集**（包含完整 `booth_cost` / `total_profit`）
- ✅ **員工 SELECT markets 表 = 0 row**（必須走 `staff_accessible_markets` view）
- ⚠️ **`is_collaborative = true` 市集**只有事實 owner 看得到（多 owner 場景）

**is_collaborative 場景處理**（需後續驗證）：

- 從 002 L82-85 看，market_created event 會**只為 actor_id 加一行 owner**
- 若「is_collaborative 市集有多個 owner」，C2.29B-2.1 後**只有事實 owner 透過底表看得到**；其他 owner 必須走 view
- → **需驗證**：既有 is_collaborative UX 是否會破壞
- 若破壞：C2.29B-2.1 改用 `mm.user_id + mm.role = 'owner'` 取代 `m.owner_id`

#### 3.2.2 `products` SELECT 政策

**現狀**（035 L208 + 014「Users can view own and team products」）：

```sql
-- 014 (推測，沒看到原始碼)
CREATE POLICY "Users can view own and team products"
ON products FOR SELECT
TO authenticated
USING (
  owner_id = auth.uid()
  OR market_id IN (SELECT * FROM public.current_user_market_ids())
);
```

**C2.29B-2.1 規劃**：

```sql

-- Owner branch: 直接從 products.owner_id 判斷
CREATE POLICY "products_select_owner_only"
ON public.products FOR SELECT
TO authenticated
USING (owner_id = auth.uid());

-- Staff branch: 拒絕（員工必須走 view）
```

**為何這樣改**：

- ✅ `products.owner_id` 已是 owner 的事實欄位（沒 trigger 維護問題）
- ✅ **員工 SELECT products 表 = 0 row**（必須走 `staff_accessible_products` view）

#### 3.2.3 `events` SELECT 政策

**現狀**（005 L53-64）：

```sql
CREATE POLICY "用戶可以查看自己市集的事件_v2"
ON events FOR SELECT
TO authenticated
USING (
  market_id IS NULL OR
  market_id IN (
    SELECT m.market_id 
    FROM market_members m 
    WHERE m.user_id = auth.uid()
  )
);
```

**C2.29B-2.1 規劃**：

**方案 A（推薦）**：直接拒絕員工 SELECT events 表

```sql
DROP POLICY "用戶可以查看自己市集的事件_v2" ON public.events;

-- Owner branch: 直接從 events.actor_id 判斷
-- 自己的事件：actor_id = auth.uid()
-- 自己市集的事件：JOIN markets m ON m.id = e.market_id WHERE m.owner_id = auth.uid()
CREATE POLICY "events_select_owner_only"
ON events FOR SELECT
TO authenticated
USING (
  actor_id = auth.uid()                              -- 自己的所有事件
  OR market_id IN (
    SELECT id FROM public.markets WHERE owner_id = auth.uid()  -- 自己市集的事件
  )
);

-- Staff branch: 拒絕
```

**方案 B（保守）**：保留員工 SELECT，但加觸發器脫敏

- 員工仍可 SELECT events 表
- 但 **BEFORE SELECT 觸發器**把敏感 payload 移除
- 風險：Supabase PostgREST 對觸發器支援有限
- → **不採**（用方案 A）

**為何選方案 A**：

- ✅ **員工 SELECT events 表 = 0 row**（必須走 `staff_accessible_events` view）
- ✅ Owner 仍可 SELECT 自己所有事件 + 自己市集的所有事件
- ⚠️ **刪除 service / trigger 仍可寫入 events**（不受 SELECT 政策影響）

### 4.3 C2.29B-2.2 的真正目標

**不是為了「員工誤觸底表」**（這是 DevTools 攻擊）— **而是「防止未來新增的程式碼不小心讓員工走底表」**。

**規劃策略**：

#### 4.3.1 方案 A（推薦）：建立 `lib/supabase/typed-client.ts` 類型守衛

```typescript
// lib/supabase/typed-client.ts (規劃)
import { createClient } from '@supabase/supabase-js';
import { useUserRole } from '@/hooks/useUserRole';

type Tables = {
  // 員工可走
  'staff_accessible_markets': true;
  'staff_accessible_products': true;
  'staff_accessible_events': true;
  // 員工不可走（底表）
  'markets': false;  // type-level: 員工 session 編譯期報錯
  'products': false;
  'events': false;
  // 雙方可走（無敏感）
  'market_members': true;
  'profiles': true;
};

type Client = {
  [K in keyof Tables]: Tables[K] extends true
    ? <T = any>(query: string) => QueryBuilder<T>
    : never;  // 員工 session 編譯期報錯
};

export function createTypedClient(userId: string, isStaff: boolean): Client {
  if (isStaff) {
    return createStaffClient();  // 員工 client：拒絕 markets / products / events
  }
  return createOwnerClient();
}
```

**優點**：

- ✅ **編譯期阻擋**員工 session 走底表
- ✅ TypeScript 100% 涵蓋（不依賴 runtime check）
- ✅ 既有 supabase client 用法可平滑遷移

**缺點**：

- ⚠️ 改動範圍大（需重構 `lib/supabase/*`）
- ⚠️ Owner service code 也要分流（`recovery.ts` 等）
- ⚠️ **DevTools 攻擊仍存在**（攻擊者改程式碼繞過 type check）— 但 RLS 已是 Sub-Phase 2.1 阻擋

#### 4.3.2 方案 B（保守）：加 runtime check + 測試

```typescript
// lib/supabase/staff-guard.ts (規劃)
import { isStaff } from '@/lib/permissions/check';

const STAFF_FORBIDDEN_TABLES = ['markets', 'products', 'events'];

export function assertNotStaffTable(table: string) {
  if (isStaff() && STAFF_FORBIDDEN_TABLES.includes(table)) {
    throw new Error(
      `[staff-guard] Staff cannot query ${table} directly. Use staff_accessible_${table} view instead.`
    );
  }
}
```

- ✅ 改動小（加在每個 supabase.from() 前）

- ⚠️ 依賴 `isStaff()` 正確性
