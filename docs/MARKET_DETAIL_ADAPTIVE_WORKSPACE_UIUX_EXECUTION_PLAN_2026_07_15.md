## 1. 最終目標

把市集詳情從「一次展開所有功能的長頁面」改為「依角色與市集階段選擇預設任務的工作台」。功能完整保留，但只有當下最重要的內容位於第一層，讓現場成交更快、收班更清楚、分析更容易閱讀。

成功後應達成：

- 營業中進入頁面時，交易工作台是第一個主要操作。
- 市集前後不再由交易區占據視線，改以準備任務或結果摘要為主。
- 老闆與員工看到符合工作責任的資訊，不只是不完整地隱藏敏感數字。
- 營收、成交、互動與時間資訊各自只有一個主要摘要來源。
- 所有既有功能都能在三次點擊內抵達，且不需要在長頁面中搜尋。
- 階段改變不會強制切換正在操作的視圖。

## 2. 設計原則

1. **角色決定內容**：老闆關注經營結果與設定；員工關注交易、紀錄與任務。
2. **階段決定預設**：市集前、營業中與市集後只影響首次進入的預設視圖。
3. **使用者保有控制**：使用者切換後，系統不因時鐘跨越營業時間而跳頁。
4. **先摘要再深入**：先顯示最多三個 KPI，再顯示一張趨勢，最後才是詳細資料。
5. **一個畫面一個主任務**：現場是成交，總覽是理解結果，管理是準備與設定。
6. **穩定核心零侵入**：重排只改 React 組合與顯示條件，不改交易及照片事件處理器。

## 5. 分階段執行

### Phase 1：重新編排與導覽

工作項目：

1. 建立共用且可鍵盤操作的三段式導覽。
2. 依階段設定老闆與員工的初始視圖。
3. 把既有元件移入角色對應視圖，不改事件處理、資料來源或 Dialog。
4. 現場視圖只保留互動、交易、最近紀錄及待補照片入口。
5. 將狀態、照片規則、成本、設備與刪除移至管理／任務。

驗收：交易與照片 smoke test 行為完全不變；切換視圖不會寫入資料；重新進入頁面才依階段選擇預設視圖。

### Phase 2：統計去重與漸進揭露

工作項目：

1. 總覽只保留收入、淨利與成交三項摘要。
2. 每日表現改為緊湊列表，移除互動明細與重複總計。
3. 流水帳支援「最近數筆」與完整紀錄兩種呈現。
4. 顧客互動、成交照片及成本各自成為按需顯示的詳細視圖。
5. 移除持續顯示的操作提示卡與員工身份說明卡。

驗收：同一視圖內不重複顯示相同 KPI；詳細資料只在使用者選擇後掛載。

### Phase 3：視覺與手機精修

工作項目：

1. 減少大圓角、厚陰影和彩色統計塊，改用區段、邊框與留白建立層級。
2. 導覽在捲動時保持可見，觸控目標至少 44 px。
3. 360 px 寬度不產生水平捲動、文字重疊或按鈕變形。
4. 平板以上才啟用雙欄，交易與主要操作不被次要資訊擠壓。
5. 完成 focus、tab semantics、可見焦點與 reduced-motion 檢查。

驗收：360 x 640、390 x 844、768 x 1024 及 1440 x 900 畫面均無重疊；鍵盤可切換全部主視圖。

## 6. 穩定性護欄

- 不修改 `TransactionWorkspace` 的成交提交與照片結果 callback。
- 不修改 `useSalesPhotoEvidenceFlow`、上傳 API 或 pending queue。
- 不改角色 capability 判斷，只重新安排已允許的功能。
- 不建立第二份營收或成交狀態，摘要沿用現有 projection hook。
- 切換分頁只使用本機 React state，不寫 Dexie 或 Supabase。
- Dialog 保持掛在頁面根層，不放入會因切換而被意外截斷的容器。
