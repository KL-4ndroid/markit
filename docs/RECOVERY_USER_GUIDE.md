# Recovery User Guide

**適用頁面：** `/recovery`
**適用對象：** 一般使用者、開發者

---

## 1. 何時需要進入 /recovery

當你遇到以下任一情況時，應前往 `/recovery` 頁面：

- 進入市集或商品頁面時，畫面長時間停留在載入狀態
- 資料（例如營業額、成交筆數）顯示為 `NaN` 或 `-`
- 匯入備份檔後部分資料消失或不完整
- 系統出現「資料庫初始化失敗」或類似錯誤提示
- 同步功能（多人協作）異常，且重新整理後問題仍在

---

## 2. 點擊「備份」後應保存備份檔

### 為什麼要先備份

`/recovery` 頁面的「備份」功能會將你目前的 IndexedDB 資料匯出為一個 `.json` 壓縮檔（`.gz`）。

**在執行任何修復、清除、或匯入操作之前，強烈建議先完成備份。**

### 操作步驟

1. 點擊「備份」或「匯出資料」
2. 瀏覽器會下載一個檔案，命名格式類似：`markit-backup-2026-05-26.json.gz`
3. 將這個檔案**保存到雲端硬碟或隨身碟**，不要只存在預設下載資料夾
4. 建議將檔案命名為有意義的名稱，方便日後比對：`markit-backup-[日期]-[問題描述].json.gz`

### 備份涵蓋的資料

- 所有市集記錄（名稱、日期、狀態、統計）
- 所有商品資料（名稱、單價、庫存）
- 所有交易與互動事件
- 每日統計資料
- 系統設定

---

## 3. 點擊「檢查」出現 dailyStats cost 無效代表什麼

### 症狀

執行檢查後，出現訊息如：

```
dailyStats cost 無效：NaN 或負數
```

或控制台出現紅色警告。

### 代表的意思

`dailyStats` 是「每日統計」的縮寫。正常情況下，每日的成本（cost）應為 0 或正數。

「cost 無效」代表你的本地資料庫中，**某一筆每日統計的成本被記錄為 `NaN`（非數字）或負數**，這通常是以下原因造成：

- 過往刪除成交記錄時，計算邏輯有誤
- 匯入的備份檔來自不同 schema 版本
- 早期版本程式碼的 bug

### 這會影響什麼

成本為 `NaN` 或負數不會讓系統崩潰，但可能導致：
- 該日營業額統計顯示異常
- 毛利潤（profit = revenue - cost）計算錯誤
- 分析圖表出現不合理數值

### 需要擔心嗎

**不需要太擔心。** 這個問題可以安全修復，且修復後不會影響其他正常資料。

## 4. 一般使用者如何安全排除

### 建議操作順序

```
步驟 1：備份（必做）
  ↓
步驟 2：檢查
  ↓
步驟 3：修復（如果檢查有發現問題）
  ↓
步驟 4：再次檢查
  ↓
步驟 5：重整（重建統計快照）
```

### 詳細說明

#### 步驟 1：備份

每次操作前先備份。備份不修改任何資料，絕對安全。

#### 步驟 2：檢查

點擊「檢查」或「驗證資料完整性」。

- 如果**沒有顯示錯誤**：表示資料目前沒有發現已知的結構性問題，可以放心繼續使用。
- 如果**有顯示錯誤**：記下錯誤訊息，繼續步驟 3。

#### 步驟 3：修復

常見的「修復」功能會自動修正以下問題：

- `NaN` 或負數的成本 → 修正為 `0`
- 空白或無效的商品 ID → 從統計中移除該筆記錄
- 損壞的每日統計 → 嘗試還原為合理數值

**修復是安全的**，因為：
- 原始事件記錄（events 表）不會被修改
- 只有計算出來的「快照」（snapshot）資料會被修正
- 日後可以隨時從事件重放重建這些統計

#### 步驟 4：再次檢查

修復後，再次點擊「檢查」，確認錯誤已消除。

#### 步驟 5：重整（重建快照）

如果檢查仍然有問題，或你懷疑統計數字與實際事件不符，可以執行「重建快照」。

這會：
1. 清空目前的所有統計快照
2. 從頭重新播放所有歷史事件
3. 重新計算每個市集、每個商品的統計數字

**重建快照可能需要幾秒到幾十秒，取決於你的事件記錄數量。**

## 5. 開發者如何進一步排查

### 查看瀏覽器控制台

在 `/recovery` 頁面按下 `F12` 開啟開發者工具，切換到「Console」分頁。

常見的關鍵字：

| 關鍵字 | 意義 |
|---|---|
| `dailyStats` | 每日統計資料 |
| `productsSold` | 商品銷售明細 |
| `deal_closed` | 成交事件 |
| `deal_deleted` | 刪除成交事件（tombstone） |
| `IntegrityError` | 資料完整性錯誤 |
| `repair` | 修復相關訊息 |

### 匯出後手動檢查

1. 下載 `.json.gz` 備份檔後，可用任何文字編輯器開啟（需先解壓縮）
2. 搜尋 `productsSold` 區塊，檢查是否有異常數值
3. 搜尋 `cost` / `revenue` 欄位，確認無 NaN

### 確認事件時間軸

如果懷疑某筆統計異常，可以：

1. 在控制台執行 `queryEvents({ type: 'deal_closed' })` 查看所有成交事件
2. 對比每日統計中的 `revenue` 與事件中的 `totalAmount` 是否相符
3. 如果不相符，重建快照通常可以修正

## 6. 什麼情況不要自行修復，應先備份

以下情況在點擊任何「清除」或「重設」按鈕之前，**請務必先聯繫開發者或確認已有備份**：

### 警示訊號

- 頁面載入後出現「初始化失敗」且無「修復」選項
- 點擊「檢查」後出現大量錯誤，涵蓋多個資料表（市集、商品、統計同時出錯）
- 匯入備份後，出現「版本不相容」警告
- 同步功能中斷後，本地資料與雲端資料出現明顯不一致

### 為什麼要先備份

這些情況可能代表：
- 資料庫 schema 已變更但匯入時未正確轉換
- 同步衝突導致資料損壞
- 嚴重的程式碼 bug

在這些情況下，**「清除資料」或「重設」可能會永久刪除無法還原的資料**。有完整備份後，才能安全嘗試修復。

## 7. 建議操作順序：備份 → 檢查 → 修復 → 再檢查 → 重整

```
┌─────────────────────────────────────────────────────┐
│  1. 備份（必做）                                    │
│     「備份」按鈕 → 儲存到雲端                       │
└─────────────────────────┬───────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────┐
│  2. 檢查                                            │
│     「檢查」按鈕 → 觀察錯誤訊息                      │
└─────────────────────────┬───────────────────────────┘
                          ↓
                    有錯誤？
                    ┌────┴────┐
                    Yes       No
                    ↓         ↓
┌─────────────────────────┐   結束（正常）
│  3. 修復                │
│     「修復」按鈕         │
└─────────┬───────────────┘
          ↓
┌─────────────────────────────────────────────────────┐
│  4. 再檢查                                         │
│     「檢查」按鈕 → 確認錯誤已消除                   │
└─────────────────────────┬───────────────────────────┘
                          ↓
                    仍有錯誤？
                    ┌────┴────┐
                    Yes       No
                    ↓         ↓
┌─────────────────────────┐   結束（修復成功）
│  5. 重整                │
│     「重建快照」按鈕     │
│     （可能需要幾十秒）    │
└─────────────────────────┘
```

## 8. 常見錯誤與對應處理

| 錯誤訊息 | 意義 | 處理方式 |
|---|---|---|
| `dailyStats cost 無效：NaN` | 某日成本為非數字 | 執行「修復」，然後重新檢查 |
| `dailyStats cost 無效：負數` | 某日成本為負數 | 執行「修復」，然後重新檢查 |
| `productsSold 有空白 productId` | 統計中有無效商品 ID | 執行「修復」移除該筆記錄 |
| `backup version 不支援` | 備份檔來自過舊版本 | 聯繫開發者，不建議自行匯入 |
| `replay before target event` | 刪除事件的時間早於目標事件 | 屬於已知限制，資料統計可能不精確；建議重建快照 |
| `duplicate tombstone` | 同一事件被刪除兩次 | 無需處理，系統已自動忽略 |
| `market_id 缺失` | 市集事件缺少關聯的市集 ID | 重建快照或聯繫開發者 |
| 初始化失敗 | IndexedDB 無法開啟 | 嘗試使用不同瀏覽器，或聯繫開發者 |
