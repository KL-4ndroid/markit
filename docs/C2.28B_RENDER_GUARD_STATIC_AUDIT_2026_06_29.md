# C2.28B Render Guard / Role Fail-Closed Static Audit

Date: 2026-06-29
Scope: static audit only
Risk: low

## Safety Boundary

This audit did not change runtime behavior.

No changes were made to:

- role capability semantics;
- Supabase migrations or RLS policies;
- sync routing;
- cache replacement execution;
- local cache deletion or overwrite behavior;
- market/product/revenue/inventory projections.

## Audited Surfaces

### Role Resolution

Files:

- `lib/permissions/role-fail-closed.ts`
- `hooks/useUserRole.ts`

Result: passed.

Evidence:

- `deriveRolePermissions()` only grants owner permissions when role loading is finished, no role error exists, and `userRole` is resolved.
- `deriveSafeInfoLevel()` returns `0` for loading, role errors, and unresolved user role.
- `useUserRole()` fail-closed catch sets a staff-like denied role instead of falling back to owner.
- `useUserRole()` returns `isOwner`, `canEdit`, and `canViewSensitiveData` from derived fail-closed permissions.
- Async role load commits are guarded by mounted state, request id, and current user id.

### App Render Guard

Files:

- `components/auth/RoleGuard.tsx`
- `components/auth/RoleLoadingFallback.tsx`

Result: passed.

Evidence:

- Protected routes render `RoleLoadingFallback` while role is loading or role lookup has errored.
- Public routes remain excluded from role lookup.
- Protected page content is not rendered until role resolution succeeds.

### Navigation Guard

File:

- `components/BottomNavigation.tsx`

Result: passed.

Evidence:

- `analytics` is treated as blocked when the user is staff or role state is unresolved.
- The click handler prevents navigation in the same staff/unresolved states.

### Sync Visibility Guard

File:

- `lib/sync-context.tsx`

Result: passed.

Evidence:

- `SyncProvider` derives `safeInfoLevel` through `deriveSafeInfoLevel()`.
- `useSync()` receives the safe info level.
- `isDataSanitized` is true unless `safeInfoLevel` is owner-level `3`.

### Market Detail Fallback Guard

File:

- `lib/markets/detail-fallback.ts`

Result: passed.

Evidence:

- Supabase fallback is blocked when staff status is pending.
- Supabase fallback is blocked for active staff mode.
- Supabase fallback is blocked when unauthenticated.

### Owner-Only Recovery And Diagnostics

Files:

- `app/recovery/page.tsx`
- `components/common/OwnerPendingOperationDiagnosticsPanel.tsx`
- `components/common/OwnerRevenueGapRepairPanel.tsx`
- `components/common/LocalProjectionRepairPanel.tsx`

Result: passed.

Evidence:

- `/recovery` derives `canUseRepairTools` from role capabilities and blocks before rendering repair panels.
- Owner pending operation diagnostics blocks unauthenticated, loading, and staff users.
- Retry/recovery handlers re-check owner-only state before action.
- Revenue and projection repair panels also block loading and staff states locally.

### Direct Route Staff Detail Guards

Files:

- `app/markets/[id]/page.tsx`
- `components/markets/StaffMarketDetailView.tsx`
- `app/products/[id]/page.tsx`

Result: passed through existing route guard tests.

Evidence:

- Staff market detail direct route returns `StaffMarketDetailView` before owner-only controls.
- Staff product detail direct route hides owner-sensitive product finance and blocks delete actions.
- Manager edit controls remain capability-gated.
