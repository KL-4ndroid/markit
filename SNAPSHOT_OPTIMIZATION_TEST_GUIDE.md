## 🧪 測試案例

### 測試 1：手動生成快照 ⭐ 核心功能

**目的**: 驗證快照可以成功生成並上傳

**步驟**:
1. 使用帳號 A 登入
2. 確保有一些數據（至少 10 個市集）
3. 進入「設定」頁面
4. 展開「資料庫管理」
5. 點擊「生成快照」按鈕

**預期結果**:
- ✅ 顯示 loading 提示「正在生成快照...」
- ✅ 控制台輸出：
  ```
  📸 開始生成快照...
  📊 讀取數據庫...
  📦 原始數據大小: X KB
  🗜️ 壓縮數據...
  ✅ 壓縮完成: Y KB (節省 Z%)
  ☁️ 上傳到雲端...
  ✅ 快照已生成：N 個事件，耗時 Xms
  ```
- ✅ 顯示成功提示「快照已生成」
- ✅ 壓縮比例約 60-80%

**驗證**:
```sql
-- 在 Supabase SQL Editor 執行
SELECT 
  id,
  event_count,
  data_size_bytes,
  compressed_size_bytes,
  compression_ratio,
  snapshot_at
FROM snapshots
WHERE user_id = 'YOUR_USER_ID'
ORDER BY snapshot_at DESC
LIMIT 1;
```

**目的**: 驗證達到閾值時自動生成快照

  ```
  ✅ 快照已生成
  ```
- ✅ 不阻塞用戶操作
- ✅ 在 Supabase 可以看到新快照

### 測試 5：壓縮效果驗證

**目的**: 驗證壓縮比例達到預期

**步驟**:
1. 生成快照
2. 在 Supabase 查看快照記錄

**SQL 查詢**:
```sql
SELECT 
  event_count,
  data_size_bytes,
  compressed_size_bytes,
  compression_ratio,
  ROUND((data_size_bytes::numeric / 1024), 2) as original_kb,
  ROUND((compressed_size_bytes::numeric / 1024), 2) as compressed_kb
FROM snapshots
WHERE user_id = 'YOUR_USER_ID'
ORDER BY snapshot_at DESC
LIMIT 5;
```

**預期結果**:
- ✅ compression_ratio 在 60-80 之間
- ✅ compressed_size_bytes 明顯小於 data_size_bytes
- ✅ 壓縮後大小約為原始大小的 20-40%
